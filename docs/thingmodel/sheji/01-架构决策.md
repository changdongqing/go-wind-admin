# 01 · 架构决策

## 1. 背景

《物模型属性单位清单》定义了建筑机电设备物模型所需的全量计量单位体系：**40+ 物理量分类、220+ 单位**，涵盖温度、压力、长度、面积、体积、质量、时间、速度、电压、电流、功率、能量、频率、湿度、照度、角度、浓度、流量、力、力矩、电阻、电容、电感、声级、数据速率、数据大小、电池、粘度、密度、加速度、热学、磁学、无量纲、计数等。

物模型属性（Property）在定义时必须绑定单位（如"供水温度 ℃""额定功率 kW""累计能耗 kWh"）。当前系统**没有任何单位管理能力**——物模型域（`thingmodel`）尚未建立。本设计负责在此空白之上，落地"模型属性单位管理"作为物模型体系的**第一块基石**。

## 2. 核心决策：模块化，而非插件化

### 2.1 判定结论

> **本功能以"新的领域模块"形态落地，不采用插件化。**

### 2.2 判定依据（基于代码实际，非假设）

逐层核查项目当前架构，**未发现任何插件化机制**：

| 层 | 实际架构 | 是否插件化 | 证据 |
|----|----------|-----------|------|
| 后端整体 | Kratos **单体多模块**（"当前 admin 服务为单体架构，但采用了微服务框架搭建"——`backend/CLAUDE.md`） | 否 | 单进程、Wire 静态依赖注入、编译期装配 |
| 后端领域划分 | 按**领域域（domain）**组织 proto 包：`identity` / `permission` / `dict` / `task` / `audit` / `storage` / `internal_message` / `authentication` | 否 | 每域一套 `protos/<domain>/service/v1/` + ent schema + repo + service，无运行时注册 |
| 依赖注入 | google/wire **代码生成**（`gow wire`），非运行时容器 | 否 | provider set 静态注册，无插件发现 |
| 前端整体 | React 19 单页应用 | 否 | 单 bundle，无 module federation / 微前端 |
| 前端模块 | `router/modules/*.tsx` 经 `import.meta.glob` **编译期**自动导入 | 否 | 编译期静态分析，非运行时动态加载 |
| 前端数据层 | `apiClient` 单例 + 懒加载 getter | 否 | 服务列表在 `client.ts` 中硬编码 |

**结论**：项目从后端到前端、从构建到运行，全程是**编译期模块化**，不存在插件注册表、动态加载器、运行时扩展点等任何插件化基础设施。强行设计成"插件"将引入一整套本项目没有的机制（注册中心、生命周期、沙箱、版本协商），成本极高、收益为零。

### 2.3 模块定位：新建 `thingmodel` 领域域

单位是物模型属性的基础字典。物模型域（Thing Model）未来将包含：

```
thingmodel（物模型领域域）
├── unit              ← 本期：模型属性单位管理（物理量分类 + 单位 + 换算）
├── property          ← 未来：物模型属性定义（引用 unit）
├── event             ← 未来：物模型事件
├── service           ← 未来：物模型服务/动作
└── product / device  ← 未来：产品（物模型模板）/ 设备实例
```

因此本期新建 `thingmodel` 领域域，在其中实现 `unit` 子能力。这与现有 `dict`（字典）、`identity`（身份）等域的划分粒度完全一致。

> **命名说明**：领域域目录用 `thingmodel`（与 `internal_message` 一样用下划线分隔多词），proto package 用 `thingmodel.service.v1`。单位子能力体现在 proto 文件名 `unit_category.proto` / `unit.proto` 上，不额外建子目录，与 `dict` 下并列 `dict_type.proto` / `dict_entry.proto` 一致。

## 3. 总体架构

完全复用现有 Contract-First 流水线与三层架构，**零新依赖**：

```

│    unit_category.proto   unit.proto   thingmodel_error.proto     │     定义 message + gRPC Service（无 http 注解）
└─────────────────────────────────────────────────────────────────┘
                              │ make api（buf generate）
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  backend/api/protos/admin/service/v1/                           │  ← BFF 层（新增）

└──────────┬───────────┴──────────────────┬───────────────────────┘
           ▼                              ▼
```

## 4. 与物模型生态的关系（前瞻）

单位管理本期作为**独立可用的基础字典**，但其设计服务于整个物模型体系：

```
                       引用 unit_id
thing_property ──────────────────────► thingmodel_unit
（未来：物模型属性）                         ▲
                                            │ 基准单位归属
                                  thingmodel_unit_category
```

- **本期**：单位 CRUD + 换算，独立成模块，不依赖任何未实现的物模型表。
- **未来挂钩点**：
  1. `thing_property.unit_id` 外键引用本模块单位；本模块提供"**引用计数**"（`reference_count` 字段或实时统计），被引用的单位**禁止物理删除**，仅允许停用。
  2. 设备实时数据上报时，按属性的 `display_unit_id` 与原始 `unit_id` 调用本模块换算 API 做归一化展示。
  3. 用户单位偏好（如"温度统一显示 ℃""压力统一显示 MPa"）可基于本模块的"基准单位 + 换算"实现全局归一。

这些挂钩点在 [02 数据模型](./02-数据模型设计.md) 中以字段/约束形式预留，但**本期不实现** Property 侧，避免过度设计。

## 5. 设计原则

1. **镜像 dict，不发明轮子**：物理量分类≈dict_type、单位≈dict_entry。表结构、proto 骨架、前端布局直接复用 dict 范式，降低评审与维护成本。
2. **换算引擎是唯一新增复杂度**：除换算外，其余均为标准 CRUD。把复杂度集中收敛到 [03 换算引擎](./03-单位换算引擎设计.md) 一处。
3. **结构化优于表达式**：可换算单位用 `factor + offset` 数值字段承载，**不**用字符串公式 + `eval`。杜绝注入、便于校验、可在 SQL 层聚合。
4. **种子数据即契约**：220+ 单位由种子脚本从清单生成，版本化管理；用户可在种子之上增删改，但种子本身是"出厂默认"。
5. **租户隔离 + 系统全局并存**：系统预置单位 `tenant_id = 0`（全局），各租户可扩展私有单位，与现有 `tenant_id` 约定一致。

## 6. 非目标（本期不做）

- 物模型属性（Property）/事件/服务的定义与维护。
- 设备实例、产品模板管理。
- 实时数据归一化运行时（仅提供换算 API，不构建流处理）。
- 单位偏好的全局下发与持久化（仅前端本地缓存）。
- 跨物理量的量纲推导（如由"长度""时间"自动合成"速度"）。
