## 2. 种子数据生成策略

### 2.1 推荐方案：Go 种子程序

新增 `backend/app/admin/service/internal/data/migrate/seed_thingmodel_unit.go`（或并入现有迁移流程），在服务首次启动/迁移时执行（幂等：按 `code` upsert，仅 `tenant_id=0`）：

```go
// 伪代码
var seedUnitCategories = []SeedCategory{
    {Code: "temperature", Name: "温度", NameEn: "Temperature", BaseUnitSymbol: "K", Quantity: "热力学温度", Units: []SeedUnit{
        {Code: "kelvin",    Symbol: "K", Name: "开尔文", IsBase: true,  Conv: LINEAR, Factor: 1,       Offset: 0},
        {Code: "celsius",   Symbol: "℃", Name: "摄氏度", IsBase: false, Conv: AFFINE, Factor: 1,       Offset: 273.15, Formula: "x+273.15"},
        {Code: "fahrenheit",Symbol: "℉", Name: "华氏度", IsBase: false, Conv: AFFINE, Factor: 5.0/9.0, Offset: 255.372, Formula: "(x+459.67)*5/9"},
        {Code: "rankine",   Symbol: "°R",Name: "兰氏度", IsBase: false, Conv: LINEAR, Factor: 5.0/9.0, Offset: 0, Formula: "x*5/9"},
    }},
    // ... 其余 40 分类
}

func SeedThingmodelUnits(ctx context.Context, client *ent.Client) error {
    for _, c := range seedUnitCategories {
        cat := upsertCategory(ctx, client, c)       // 按 code upsert，tenant_id=0
        for _, u := range c.Units {
            upsertUnit(ctx, client, cat.ID, u)      // 按 (tenant_id, code) upsert
        }
    }
    return nil
}
```

- **幂等**：以 `(tenant_id=0, code)` 为键 upsert，重复执行不报错、不产生重复。
- **版本化**：种子源码随仓库版本管理；清单更新时改种子、提 PR、重新迁移。
- **用户数据保护**：仅 upsert `tenant_id=0` 的系统预置数据；租户自建单位（`tenant_id>0`）永不被覆盖。
- **触发时机**：可接入 Kratos 启动 hook 或独立的 `gow seed` 命令；建议做成**显式命令**而非每次启动自动跑，避免生产环境意外。

### 2.2 备选方案：SQL 种子脚本

若不便写 Go seed，可生成纯 SQL（PostgreSQL）放 `backend/app/admin/service/configs/` 或 `scripts/sql/`，由部署流程执行。例：

```sql
-- 温度分类 + 单位
INSERT INTO thingmodel_unit_categories (code, name, name_en, base_unit_symbol, quantity, is_enabled, sort_order, tenant_id, created_at)
VALUES ('temperature','温度','Temperature','K','热力学温度', true, 1, 0, NOW())
ON CONFLICT (tenant_id, code) DO UPDATE SET name = EXCLUDED.name, base_unit_symbol = EXCLUDED.base_unit_symbol;

-- 单位（category_id 用子查询取 temperature 的 id）
INSERT INTO thingmodel_units (category_id, code, symbol, name, is_base, conversion_type, factor, offset, formula_expr, precision, is_si_unit, is_legal_unit, is_enabled, sort_order, tenant_id, created_at)
SELECT c.id, v.* FROM thingmodel_unit_categories c, (VALUES
  ('kelvin','K','开尔文', true,  'LINEAR', 1.0,      0.0,      'x*1',            2, true,  true),
  ('celsius','℃','摄氏度', false, 'AFFINE', 1.0,      273.15,   'x+273.15',       2, true,  true),
  ('fahrenheit','℉','华氏度', false,'AFFINE', 0.5555556,255.372,  '(x+459.67)*5/9', 2, false, false),
  ('rankine','°R','兰氏度', false, 'LINEAR', 0.5555556,0.0,      'x*5/9',          2, false, false)
) AS v(code,symbol,name,is_base,conversion_type,factor,offset,formula_expr,precision,is_si_unit,is_legal_unit)
WHERE c.code = 'temperature' AND c.tenant_id = 0
ON CONFLICT (tenant_id, code) DO UPDATE SET symbol=EXCLUDED.symbol, name=EXCLUDED.name, factor=EXCLUDED.factor, offset=EXCLUDED.offset;
```

### 2.3 factor 精度约定

- 整数比（如 5/9）存足够位数的 double；`5.0/9.0` 在 Go/PG double 下精度足够工程用途。
- 清单中科学计数法（如 `1.602176e-19`）直接存 double。
- `formula_expr` 照抄清单原始公式文本，仅供展示。

## 5. 数据治理点（落地时需决策）

清单本身存在几处归类问题，设计上已处理或需确认：

| 问题 | 处理 |
|------|------|
| "电池"分类混有电荷量（Ah/mAh/C）与电能（Wh） | 本设计拆为 `electric_charge`（基准 Ah）与 `battery_energy`（基准 Wh）两个分类。注意 C（库仑）与 Ah 换算：1 Ah = 3600 C，故 C 的 factor=1/3600 相对 Ah 基准 |
| "湿度·露点温度"基准是 K，但归湿度分类 | 露点温度本质是温度。建议将"露点温度"归入 `temperature` 分类（符号 `℃(露点)`），或保留 humidity 但其 conversion_type 标注、不参与跨单位换算。**建议归入 temperature** |
| "浓度"分类基准混合（ppm / mg/m³ / mol/L 三个互不可换算的基准） | 浓度内含多个量纲不同的子群。严格做法是按基准拆分（`concentration_fraction` ppm基准 / `concentration_mass` mg/m³基准 / `concentration_molar` mol/L基准）。本期可先保留单一分类，跨子群换算时由 conversion_type + 不同基准自然拒绝 |
| "%" 在无量纲与浓度均出现 | 分属不同分类，code 不同（`percent` vs `concentration_percent`），不冲突 |
| 对数单位 dBm→mW | 本期不做自动换算（LOGARITHMIC），仅 formula_expr 展示 `10^(x/10) mW`。未来如需，单独实现对数换算分支 |

> **建议**：P3 种子数据落地前，由架构师确认上述拆分方案（尤其电池、露点、浓度三处），再生成种子。

## 6. 风险与遗留

| 风险/遗留 | 说明 | 应对 |
|-----------|------|------|
| `reference_count` 无实际维护方 | 本期字段预留恒 0，"被引用禁止删除"暂不生效 | 待 `thing_property` 模块落地时，由其 Create/Update/Delete 维护该计数（增减），届时启用 C6 约束 |
| 条件换算（ppm↔mg/m³）不支持 | 依赖外部参数 M/T，本期不实现 | formula_expr 展示近似公式；未来可扩展 `ConvertConditional` API 接收 M/T 参数 |
| 对数单位（dB 系）不可换算 | 数学本质决定 | 类型标记排除，展示说明 |
| partial unique index 需手补 | Ent schema 难表达 | 迁移脚本手动追加（[02 章 C1](./02-数据模型设计.md)） |
| 种子数据与清单版本同步 | 清单更新需同步种子 | 种子源码版本化，清单变更走 PR |
| 单位符号含特殊字符（℃ ³ ‰ μ） | DB 字符集需 utf8mb4 | schema 已声明 `utf8mb4` / `utf8mb4_bin` |

## 7. 未来演进（非本期）

1. **物模型属性引用**：`thing_property.unit_id` → 本模块；维护 `reference_count`；属性定义时的单位选择器直接用 `ListByCategory`。
2. **设备数据归一化**：设备上报原始值 + 原始单位 → 按属性展示单位 `Convert` → 展示/存储。
3. **单位偏好**：用户/租户级"统一展示单位"偏好，基于基准单位 + 换算实现全局归一。
4. **量纲推导**（远期）：由基本量纲自动合成复合单位（如 m÷s=m/s），需引入量纲代数，复杂度高，暂不规划。
5. **国际化扩展**：若需多语种单位名（日/韩等），届时再拆 `unit_i18n` 子表（届时回归 dict 的 i18n 模式）。
