# 02 · 数据模型设计（v2.0）

> 本章定义"分类管理"的领域模型、ER 关系、`thingmodel_categories` 表结构。
> 表结构风格严格对齐已有 `thingmodel_unit_categories` 和 `dict_type`。

---

## 1. 领域模型

```
┌─────────────────────────────────────────────────────────────────┐
│                       Category（分类节点）                         │
│                                                                  │
│   • kind:       SYSTEM | SPACE | FACILITY   （未来扩展枚举）       │
│   • code:       变长，按 level：                                  │
│                   level=1 → 2 位（如 30）                         │
│                   level=2 → 4 位（如 3001）                       │
│                   level=3 → 6 位（如 300101）                     │
│                   level=4 → 8 位（如 30010100）                   │
│                 ⚠️ 绝不补 0 凑长度                                 │
│   • level:      1 (大类) | 2 (中类) | 3 (小类) | 4 (细类)         │
│   • parent_id:  父节点 ID（自引用，level=1 时为 NULL）             │
│   • name / name_en / icon / description / sort_order / is_enabled│
│   • reference_count: 被物模型/实例引用次数（预留，本期恒 0）       │
│                                                                  │
│   ⚠️ 不存 is_leaf 字段：叶子 ⇔ level == 4                          │
│   ⚠️ 不存 path 字段：父子关系由 code 截断推导                       │
└─────────────────────────────────────────────────────────────────┘
              ▲  parent_id (FK→self, ON DELETE RESTRICT)
              │
              └─── 形成 4 层树（按 kind 切分为多棵独立森林）
```

### 1.1 关键设计抉择（与 v1.0 的差异点）

| 抉择点 | v1.0 旧方案 | **v2.0 方案** | 理由 |
|--------|------------|--------------|------|
| code 长度 | 一律 8 位（大类补 0） | **按 level 变长：2/4/6/8** | 与清单对齐；唯一键无需 level；用户输入心智简单 |
| 唯一键 | `(tenant, kind, code, level)` | **`(tenant, kind, code)`** | code 长度不同自然无冲突，唯一键退化到 3 元组 |
| 叶子标记 | `is_leaf` 冗余字段 | **不存，由 `level == 4` 推导** | 减少冗余、避免不一致 |
| 父子关系存储 | `parent_id` + `path` 双冗余 | **仅 `parent_id`**（path 由 code 截断推导） | code 自带层级前缀，path 是纯冗余 |
| 业务扩展 | scope + kind 双维度 | **仅 kind 枚举**（未来追加值） | YAGNI，scope 找不到正当使用场景 |
| 国际化 | 主表存 `name`+`name_en` | 同 v1.0 | 国标分类名稳定 |
| 多租户 | `tenant_id` mixin | 同 v1.0 | 与全表一致 |

### 1.2 极简性收益

| 指标 | v1.0 | **v2.0** |
|------|------|----------|
| schema 字段数 | 16（含 path / is_leaf） | **14**（去掉 path / is_leaf） |
| 唯一索引列数 | 4（tenant, kind, code, level） | **3**（tenant, kind, code） |
| Service 维护逻辑 | 写 path、写 is_leaf、维护父 is_leaf | **不需要** |
| 节点总数 | ≈700（含 "小类 = 首细类 同 code" 两行） | **≈600**（小类和首细类天然不冲突，各一行） |

## 2. 表结构

### 2.1 `thingmodel_categories`（统一分类表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | uint32 | PK, auto_increment | 来自 `mixin.AutoIncrementId` |
| `kind` | enum | NOT NULL, immutable | 分类种类：`SYSTEM` / `SPACE` / `FACILITY`（未来追加枚举值） |
| `code` | string(8) | NOT NULL, immutable | **变长 code**：长度 ∈ {2, 4, 6, 8}，纯数字 |
| `level` | uint8 | NOT NULL | 层级：1=大类、2=中类、3=小类、4=细类（冗余存储 = code 长度 / 2） |
| `parent_id` | uint32 | optional（level=1 时 NULL） | 父节点 ID（自引用 FK） |
| `name` | string | NOT NULL | 中文名 |
| `name_en` | string | optional | 英文名 |
| `icon` | string | optional | Iconify 图标名 |
| `description` | string | optional | 描述/备注 |
| `is_enabled` | bool | default true | 来自 `mixin.IsEnabled` |
| `sort_order` | uint32 | default 0 | 来自 `mixin.SortOrder`，**默认按 code 升序即可，sort_order 仅用于人工覆盖** |
| `reference_count` | uint32 | default 0 | 被物模型/实例引用次数（预留，本期恒 0） |
| `tenant_id` | uint32 | default 0 | 来自 `mixin.TenantID`，0=系统全局 |
| `created_by/updated_by/deleted_by` | uint32 | optional | 来自 `mixin.OperatorID` |
| `created_at/updated_at/deleted_at` | timestamp | optional | 来自 `mixin.TimeAt` |

### 2.2 Ent Schema 骨架

> 风格对齐 `thingmodel_unit_categories`（见 `internal/data/ent/schema/unit_category.go`）。

```go
package schema

import (
    "entgo.io/ent"
    "entgo.io/ent/dialect/entsql"
    "entgo.io/ent/schema"
    "entgo.io/ent/schema/edge"
    "entgo.io/ent/schema/field"
    "entgo.io/ent/schema/index"

    "github.com/tx7do/go-crud/entgo/mixin"
)

// Category 物模型-分类表（智能系统/空间/设备设施 统一承载，未来可扩展 kind）
type Category struct{ ent.Schema }

func (Category) Annotations() []schema.Annotation {
    return []schema.Annotation{
        entsql.Annotation{
            Table:     "thingmodel_categories",
            Charset:   "utf8mb4",
            Collation: "utf8mb4_bin",
        },
        entsql.WithComments(true),
        schema.Comment("物模型-分类表 / Thing model category"),
    }
}

func (Category) Fields() []ent.Field {
    return []ent.Field{
        // 分类种类（不可变）；未来扩展只需在 NamedValues 追加
        field.Enum("kind").
            NamedValues(
                "System",   "SYSTEM",
                "Space",    "SPACE",
                "Facility", "FACILITY",
                // 未来：
                // "Document",    "DOCUMENT",
                // "Certificate", "CERTIFICATE",
                // "Form",        "FORM",
            ).
            Immutable().
            Comment("分类种类（不可变）/ Category kind, extensible"),

        // 变长 code（2/4/6/8 位纯数字）
        field.String("code").
            MaxLen(8).
            NotEmpty().
            Immutable().
            Optional().
            Nillable().
            Comment("分类编码（变长 2/4/6/8 位纯数字，按 level 决定长度）/ Variable-length code"),

        // 层级 1..4
        field.Uint8("level").
            Min(1).Max(4).
            Comment("层级：1=大类 2=中类 3=小类 4=细类 / Hierarchy level (1..4)"),

        // 父节点 ID（自引用）
        field.Uint32("parent_id").
            Optional().
            Nillable().
            Comment("父节点 ID（level=1 时为空）/ Parent category id"),

        // 中/英文名
        field.String("name").
            NotEmpty().Optional().Nillable().
            Comment("中文名 / Category name (zh)"),
        field.String("name_en").
            Optional().Nillable().
            Comment("英文名 / Category name (en)"),

        // 图标/描述
        field.String("icon").Optional().Nillable().
            Comment("Iconify 图标名 / Icon name"),
        field.String("description").Optional().Nillable().
            Comment("描述 / Description"),

        // 被引用次数（预留）
        field.Uint32("reference_count").Default(0).Nillable().
            Comment("被物模型/实例引用次数（预留）/ Reference count (reserved)"),
    }
}

func (Category) Mixin() []ent.Mixin {
    return []ent.Mixin{
        mixin.AutoIncrementId{},
        mixin.TimeAt{},
        mixin.OperatorID{},
        mixin.IsEnabled{},
        mixin.SortOrder{},
        mixin.TenantID[uint32]{},
    }
}

func (Category) Edges() []ent.Edge {
    return []ent.Edge{
        // 自引用：父 → 多个子；删除受限（RESTRICT），有子时禁止物理删除
        edge.To("children", Category.Type).
            From("parent").
            Field("parent_id").
            Unique().
            Annotations(entsql.Annotation{
                OnDelete: entsql.Restrict,
            }),
    }
}
```

### 2.3 索引

```go
func (Category) Indexes() []ent.Index {
    return []ent.Index{
        // ===== 唯一约束（核心）=====
        // (tenant, kind, code) 全局唯一 —— v1.0 需要 4 列，v2.0 只需 3 列
        index.Fields("tenant_id", "kind", "code").
            Unique().
            StorageKey("uix_thingmodel_cat_tenant_kind_code"),

        // ===== 树/查询索引 =====
        // 按 kind+level 查询（"取所有大类"/"取所有细类"）
        index.Fields("tenant_id", "kind", "level").
            StorageKey("idx_thingmodel_cat_tenant_kind_level"),

        // 按父节点取子节点（编辑/删除时用）
        index.Fields("tenant_id", "kind", "parent_id").
            StorageKey("idx_thingmodel_cat_tenant_kind_parent"),

        // ===== 常用筛选 =====
        index.Fields("is_enabled").
            StorageKey("idx_thingmodel_cat_is_enabled"),
        index.Fields("sort_order").
            StorageKey("idx_thingmodel_cat_sort_order"),
        index.Fields("tenant_id").
            StorageKey("idx_thingmodel_cat_tenant_id"),

        // 名称模糊搜索
        index.Fields("name").
            StorageKey("idx_thingmodel_cat_name"),

        // 子树前缀查询（按 code LIKE 'xxx%'）—— 由 (tenant_id, kind, code) 唯一索引天然支持
        // （PostgreSQL 上前缀 LIKE 可走索引）
    }
}
```

## 3. code 规则与父子推导

### 3.1 code 规则汇总

| level | code 长度 | 示例 | 来源 |
|-------|----------|------|------|
| 1 大类 | 2 位 | `30` / `10` / `20` | 清单 H2 标题"## NN ..."的 NN |
| 2 中类 | 4 位 | `3001` / `1001` / `2001` | 清单细类 code8 的前 4 位 |
| 3 小类 | 6 位 | `300101` / `100101` / `200101` | 清单细类 code8 的前 6 位 |
| 4 细类 | 8 位 | `30010100` / `30010101` / `10010100` | 清单"编码"列原文 |

### 3.2 父子推导算法（无需 path 字段）

```go
// 由子 code 求父 code：直接按 level 截断
func parentCode(self string) string {
    if len(self) <= 2 {
        return ""  // 大类无父
    }
    return self[:len(self)-2]
}

// 校验子 code 合法性：必须以父 code 为严格前缀，且恰好多 2 位
func validateChildCode(self, parent string) error {
    if !strings.HasPrefix(self, parent) {
        return errors.New("child code must start with parent code")
    }
    if len(self) != len(parent)+2 {
        return errors.New("child code length must be parent length + 2")
    }
    if _, err := strconv.Atoi(self); err != nil {
        return errors.New("code must be all digits")
    }
    return nil
}

// 由 code 推导 level
func levelOfCode(code string) uint8 {
    return uint8(len(code) / 2)
}
```

### 3.3 子树查询（不需要 path 字段）

取「冷热源系统」（中类 `3001`）整子树：

```sql
SELECT * FROM thingmodel_categories
WHERE tenant_id = 0 AND kind = 'SYSTEM'
  AND (code LIKE '3001%')
ORDER BY code ASC;
```

> 返回 `3001` 自身 + 所有 `3001xx`（小类）+ 所有 `3001xxxx`（细类），按 code 升序天然就是深度优先顺序。

唯一索引 `(tenant_id, kind, code)` 在 PostgreSQL 下**前缀 LIKE 可走索引**（默认 collation 是 C 或 ascii 时；如果 collation 是 utf8mb4_bin 也支持），性能优于 v1.0 的 `path LIKE` 方案。

## 4. 约束（应用层 + DB 层）

| 编号 | 约束 | 实现层 | 说明 |
|------|------|--------|------|
| C1 | level=1 时 parent_id 必须为 NULL；level∈{2,3,4} 时 parent_id 必填 | Service | 写入前校验 |
| C2 | parent_id 指向的节点 level 必须 = 本节点 level - 1，且 kind 与 tenant_id 一致 | Service | 防止跨层、跨 kind、跨租户挂载 |
| C3 | code 必须为纯数字字符串，长度 ∈ {2, 4, 6, 8} | Service | 正则 `^\d{2}$|^\d{4}$|^\d{6}$|^\d{8}$` 或按 `level*2` 精确匹配 |
| C4 | **code 长度必须 = level × 2** | Service | 强一致校验：level=1→len=2、level=4→len=8 |
| C5 | code 必须以父节点 code 为**严格前缀**，且**恰好**比父长 2 位 | Service | 见 §3.2 validateChildCode |
| C6 | `(tenant_id, kind, code)` 唯一 | DB 唯一索引 | 见 §2.3 |
| C7 | 删除节点时，若 children 存在则禁止 | DB `ON DELETE RESTRICT` + Service 友好提示 | 上层需自下而上删除 |
| C8 | 删除节点时，若 `reference_count > 0` 则禁止物理删除，仅允许停用 | Service | 为未来引用方预留；本期恒满足 |
| C9 | `code` 不可变（immutable）；`kind` 不可变 | Ent 字段 `Immutable()` | 程序引用的稳定标识 |
| C10 | 停用大/中/小类时，前端提示其子树将被隐藏；数据库不级联停用 | UI 层 | 查询时按 code 前缀联合 is_enabled 过滤 |

### 4.1 关于"不存在结尾为字符 0 的 code"的精确语义

- ✅ **大/中/小类 code 不补 0**：大类是 `30`，不是 `30000000`；中类是 `3001`，不是 `30010000`。
- ⚠️ **细类 code 末两位可以是 `00`**：如 `30010100`「压缩式制冷系统」、`30010200`「集中供热系统」——这是**国标规定的"该小类下第一个细类"序号**，不是"补 0 凑长度"，符合本原则。
- ✅ **没有需要在末尾补 0 的场景**：任何 code 的长度都精确等于 `level × 2`，绝无"补 0 占位"的情况。
