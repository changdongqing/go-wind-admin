## v2.0 优化原则（本版相对 v1.0 的差异）

| # | 原则 | v1.0（旧）| v2.0（本版）|
|---|------|-----------|-------------|
| 1 | **code 长度按 level 分级** | 一律补 0 凑 8 位（大类 `30000000`） | **大类 2 位、中类 4 位、小类 6 位、细类 8 位**；**绝不补 0 凑长度** |
| 2 | **叶子节点判定** | 用 `is_leaf` 冗余字段标记 | 直接由 `level == 4` 推导，**去掉 `is_leaf` 字段** |
| 3 | **业务扩展机制** | 引入 `scope` 字段 + Scope Validator | 直接**扩展 `kind` 枚举**（未来加 `DOCUMENT` / `CERTIFICATE` / `FORM`），不引入新字段 |
| 4 | **前端形态** | Tabs + Splitter（左 Tree 右 Table）+ Drawer | **Tabs（切 kind）+ 单一 ProTable + Drawer**——code 自带层级排序，无需树展示 |

## 一句话结论

本功能在已有的 `thingmodel` 领域域内新增 **`category` 子能力**：以**一张统一的分类表 + `kind` 枚举**承载"智能系统 / 空间 / 设备设施"三套国标分类（共 **3 套 ≈ 480 细类**），层级固定 **4 层**，code 长度由 level 决定（2/4/6/8）。本期**只做分类增删改查**，不涉及任何物模型属性/事件/服务的配置。

**为什么这样设计是对的**：

- code 长度按 level 分级 → **唯一键退化为 `(tenant_id, kind, code)`**（不再需要 level 参与），原 v1 中"小类 `30010100` 与细类 `30010100` 冲突"的问题**自然消失**。
- 去掉 `is_leaf` → 减少一个易于不一致的冗余字段，前端按 `level == 4` 判断即可。
- 不引入 scope → 未来加文档/证书/表单分类时，**只需 `kind` 枚举追加新值**，schema/Repo/Service 零改动。
- 前端不用树 → code 本身就是有序前缀串（`30` < `3001` < `300101` < `30010100`），ProTable 按 code 升序即天然层级；视觉缩进交给一列"层级"Tag 或 padding 即可。

## 设计目标

1. **完整覆盖三套清单**：承载智能系统（≈170 细类）、空间（≈100 细类）、设备设施（≈210 细类），合计约 **480 个细类节点**（含派生的大/中/小类，共约 **600 个分类节点**——少于 v1 的 700，因为不补 0 不再为"小类 = 细类同 code"额外建行）。
2. **强标准化**：层级**固定 4 层**、code 长度严格按 level（**2/4/6/8 位**）、与 GB/T 51269-2017、DB31/T 1320-2021、T/BCAT 0002-2022、T/CECS 1196-2022、T/CCIAT 0040-2021 对齐。
3. **可治理**：分类支持启用/停用、排序、租户隔离、引用约束（被未来 `thing_property` / `device_instance` 引用后**不可物理删除**，仅允许停用）。
4. **零侵入**：完全复用 Contract-First 流水线（proto → `make api` → `make ts`）、三层架构、`thingmodel` 域，**不引入新框架、新依赖**。
5. **可扩展**：未来要支持"文档分类 / 证书分类 / 表单分类"——**只需追加 `CategoryKind` 枚举值**（如 `DOCUMENT` / `CERTIFICATE` / `FORM`）+ 在 Service 注册对应的 KindValidator 即可，**无需新字段、无需新表、无需数据迁移**。

## 关键技术决策摘要

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 落地形态 | **领域模块**（非插件） | 全项目无插件化机制；沿用 `thingmodel` 域 |
| 数据结构 | **单表 + `kind` 枚举 + `parent_id` 自引用** | 三套清单字段完全同构；未来扩展 kind 即可承载新业务，零 schema 变更 |
| 层级模型 | **固定 4 层 + `level` 字段** | 国标本就 4 层；强约束带来更稳的种子、更直观的 UI |
| 编码模型 | **变长 code**：level 1=2 位、2=4 位、3=6 位、4=8 位，**绝不补 0** | 与清单"编码"列直接对齐（细类 8 位）；前缀截取即得父 code；唯一键无需 level 参与 |
| 唯一性 | `(tenant_id, kind, code)` 复合唯一 | code 长度按 level 不同，天然无冲突 |
| 叶子判定 | `level == 4` 即叶子，**不存 `is_leaf` 字段** | 由 level 推导，避免冗余不一致 |
| 业务扩展 | 通过 `kind` 枚举追加新值 | 不引入 scope 字段；kind 已是天然的业务域区分 |
| 前端布局 | **Tabs（切 kind）+ 单 ProTable + Drawer** | code 自带层级顺序，无需 Tree；简洁即正义 |
| 软删除 / 级联 | `On Delete: Restrict` + 应用层 has-children 校验 | 父节点有子节点时禁止物理删除 |

## 与"单位管理"的同构关系

| 维度 | 单位管理 | 分类管理（v2） |
|------|----------|----------------|
| 领域域 | `thingmodel` | `thingmodel`（同域） |
| 主资源 | UnitCategory + Unit（双表） | Category（单表 + kind 区分多业务） |
| Proto | `unit_category.proto` + `unit.proto` | `category.proto`（单文件） |
| 前端 | Splitter（左分类列表，右单位列表） | **Tabs + 单 ProTable**（更简单） |
| 种子 | 41 分类 / ≈225 单位 | 3 kind / ≈600 分类节点 |
| 引擎复杂度 | 换算引擎（独立 1 章） | 无引擎，纯 CRUD |

## 与物模型生态的关系（前瞻）

```
                                         category_id (FK)
thing_property ──────────────────────────────────────────────► thingmodel_categories
device_instance / space_instance / system_instance ──────────►        ▲
                                                              (kind 决定可挂载的细类范围)
```

- **本期**：分类独立可用——只做增删改查、启停、排序、租户隔离。
- **未来挂钩点**：引用方按 kind 限定可选分类（如 `device_instance.category_id` 只接受 `kind=FACILITY AND level=4` 的节点），并维护 `reference_count`。

- 物模型属性（Property）/事件/服务的定义与维护。
