## 1. 三套清单数据规模

| 清单 | kind | 大类数 | 中类数 | 小类数 | 细类数 | **节点总数** |
|------|------|--------|--------|--------|--------|--------------|
| `智能系统分类清单.md` | `SYSTEM` | 7（30/31/32/33/34/35/36） | ≈30 | ≈80 | ≈170 | **≈290** |
| `空间分类清单.md` | `SPACE` | 1（10） | 5 | ≈25 | ≈100 | **≈130** |
| `设备设施分类清单.md` | `FACILITY` | 7（20/21/22/23/24/25/26） | ≈30 | ≈80 | ≈210 | **≈330** |
| **合计** | — | 15 | ≈65 | ≈180 | **≈480** | **≈740** |

> ⚠️ **v2 与 v1 节点数对比**：
> - v1 因为"小类 8 位 code 与首细类 8 位 code 相同"需要靠 level 区分，实际是 **同 code 两行**。
> - v2 由于 code 变长（小类是 6 位、细类是 8 位），不存在 code 重复，**节点数稍微多一点**，但**唯一键查询逻辑简单**。
> - 实际数字差异不大，仍在 600–750 节点范围内。

| 编码     | 大类       | 中类       | 小类     | 细类             |

## 3. 解析与派生算法（v2 关键改动）

```
1. 按 "## NN 标题" 切段，识别每个大类 (level=1)：
   • code = NN（直接取 2 位，不补 0）
   • name = 标题去掉编号

2. 解析表格，每行得到 (code8, 大类名, 中类名, 小类名, 细类名)

3. 由 code8 截断派生：
   • 大类 code = code8[0:2]    （2 位，如 "30"）
   • 中类 code = code8[0:4]    （4 位，如 "3001"）
   • 小类 code = code8[0:6]    （6 位，如 "300101"）
   • 细类 code = code8         （8 位，如 "30010100"）

4. 去重：用 (kind, code) 作为唯一键（v1 是 (kind, code, level)），构建一个 map

5. 4 层节点全部入 map 后，按 code 字符串升序排序（天然保证 level 升序），
   逐条 upsert（先 level=1 大类，再 level=2，再 level=3，再 level=4，
   保证父节点先于子节点入库）

6. 每条记录的 parent_id：通过查询 (kind, parent_code) 取得（parent_code = self_code[:-2]）
```

### 3.1 关键差异（v2 vs v1）

| 步骤 | v1.0 | **v2.0** |
|------|------|----------|
| 大类 code | `"30" + "000000"` 补 0 凑 8 位 | **直接用 `"30"`** |
| 中类 code | `code8[0:4] + "0000"` 补 0 | **`code8[0:4]`**（取前 4 位） |
| 小类 code | `code8[0:6] + "00"` 补 0 | **`code8[0:6]`**（取前 6 位） |
| 唯一键 | `(kind, code, level)` | **`(kind, code)`** |
| path 字段 | 需维护拼接 | **不需要**（直接由 code 推导） |

→ 解析逻辑更简单、入库后数据更干净。

## 4. 种子数据生成策略

### 4.1 推荐方案：Go 种子程序 + 嵌入清单 Markdown

新增 `backend/app/admin/service/internal/data/migrate/seed_thingmodel_category.go`：

```go
package migrate

import (
    "context"
    _ "embed"
    "fmt"
    "regexp"
    "sort"
    "strings"
    "time"

    "go-wind-admin/app/admin/service/internal/data/ent"
    "go-wind-admin/app/admin/service/internal/data/ent/category"
    thingmodelV1 "go-wind-admin/api/gen/go/thingmodel/service/v1"
)

//go:embed seeds/智能系统分类清单.md
var systemMd string

//go:embed seeds/空间分类清单.md
var spaceMd string

//go:embed seeds/设备设施分类清单.md
var facilityMd string

type seedNode struct {
    Kind  thingmodelV1.CategoryKind
    Code  string  // 变长 2/4/6/8
    Level uint32
    Name  string
}

func SeedThingmodelCategories(ctx context.Context, client *ent.Client) error {
    nodes := make(map[string]*seedNode, 800) // key = kind|code

    addAll(nodes, parseClassListMarkdown(systemMd,   thingmodelV1.CategoryKind_SYSTEM))
    addAll(nodes, parseClassListMarkdown(spaceMd,    thingmodelV1.CategoryKind_SPACE))
    addAll(nodes, parseClassListMarkdown(facilityMd, thingmodelV1.CategoryKind_FACILITY))

    // 按 (kind, code) 字符串排序：天然 level 升序（短 code 在前）
    ordered := sortByKindAndCode(nodes)

    return upsertNodes(ctx, client, ordered)
}

func upsertNodes(ctx context.Context, client *ent.Client, nodes []*seedNode) error {
    // 缓存已 upsert 节点 (kind|code) → id，便于子节点查父 id
    idByKey := make(map[string]uint32, len(nodes))
    for _, n := range nodes {
        parentID := uint32(0)
        if n.Level > 1 {
            parentCode := n.Code[:len(n.Code)-2]
            pkey := keyOf(n.Kind, parentCode)
            parentID = idByKey[pkey]
            if parentID == 0 {
                return fmt.Errorf("parent not seeded yet: %s/%s", n.Kind, parentCode)
            }
        }

        id, err := client.Category.Create().
            SetKind(category.Kind(n.Kind.String())).
            SetCode(n.Code).
            SetLevel(uint8(n.Level)).
            SetNillableParentID(nillableIf(parentID != 0, parentID)).
            SetName(n.Name).
            SetIsEnabled(true).
            SetTenantID(0).
            SetCreatedAt(time.Now()).
            OnConflictColumns("tenant_id", "kind", "code").
            UpdateName().
            ID(ctx)
        if err != nil {
            return fmt.Errorf("upsert %s/%s/L%d: %w", n.Kind, n.Code, n.Level, err)
        }
        idByKey[keyOf(n.Kind, n.Code)] = id
    }
    return nil
}

func keyOf(kind thingmodelV1.CategoryKind, code string) string {
    return kind.String() + "|" + code
}
```

**关键点（与 v1 差异）**：

- **`OnConflictColumns("tenant_id", "kind", "code")`**：v2 唯一键只有 3 列，比 v1 少一个 level 列。
- **不维护 path / is_leaf**：上面 Create 语句中没有 `SetPath` / `SetIsLeaf`，因为字段不存在。
- **`parent_code = self_code[:-2]`**：直接字符串截断，比 v1 的"补 0 计算 + 反查"更直观。

### 4.2 解析算法细节

```go
func parseClassListMarkdown(raw string, kind thingmodelV1.CategoryKind) []*seedNode {
    h2Regex  := regexp.MustCompile(`^##\s+(\d{2})\s+(.+?)\s*$`)
    rowRegex := regexp.MustCompile(`^\|\s*(\d{8})\s*\|\s*([^|]+?)\s*\|\s*([^|]+?)\s*\|\s*([^|]+?)\s*\|\s*([^|]+?)\s*\|`)

    // (kind, code) → name；用 map 去重
    seen := map[string]string{}
    addOnce := func(code, name string) {
        if _, ok := seen[code]; !ok {
            seen[code] = strings.TrimSpace(name)
        }
    }

    var currentBigCode, currentBigName string
    for _, line := range strings.Split(raw, "\n") {
        if m := h2Regex.FindStringSubmatch(line); m != nil {
            currentBigCode, currentBigName = m[1], m[2]
            addOnce(currentBigCode, currentBigName) // 大类（2 位）
            continue
        }
        m := rowRegex.FindStringSubmatch(line)
        if m == nil {
            continue
        }
        code8 := m[1]
        // big := m[2]  —— 与 H2 标题一致，已添加；忽略
        mid, small, leaf := m[3], m[4], m[5]

        bigCode   := code8[:2]
        midCode   := code8[:4]
        smallCode := code8[:6]

        addOnce(bigCode, currentBigName) // 容错：表格行的大类列可能与 H2 略有差异，以 H2 为准
        addOnce(midCode, mid)
        addOnce(smallCode, small)
        addOnce(code8, leaf)
    }

    var nodes []*seedNode
    for code, name := range seen {
        nodes = append(nodes, &seedNode{
            Kind: kind, Code: code, Level: uint32(len(code) / 2), Name: name,
        })
    }
    return nodes
}

func sortByKindAndCode(m map[string]*seedNode) []*seedNode {
    out := make([]*seedNode, 0, len(m))
    for _, n := range m {
        out = append(out, n)
    }
    sort.Slice(out, func(i, j int) bool {
        if out[i].Kind != out[j].Kind {
            return out[i].Kind < out[j].Kind
        }
        // 关键：先按 level（=len/2），再按 code 字符串
        if out[i].Level != out[j].Level {
            return out[i].Level < out[j].Level
        }
        return out[i].Code < out[j].Code
    })
    return out
}
```

**复杂度**：O(L)，L 为清单总行数（≈1500），单次解析 < 50ms。

### 4.3 备选方案：SQL 种子脚本

由 Go 离线生成 `scripts/sql/seed_thingmodel_category.sql`，由部署流程执行：

```sql
-- 大类（level=1，code 2 位）
INSERT INTO thingmodel_categories (kind, code, level, parent_id, name, name_en, is_enabled, sort_order, tenant_id, created_at)
VALUES
  ('SYSTEM', '30', 1, NULL, '暖通空调系统', 'HVAC System', true, 30, 0, NOW()),
  ('SYSTEM', '31', 1, NULL, '给排水系统', 'Plumbing System', true, 31, 0, NOW()),
  -- ... 其余大类 ...
ON CONFLICT (tenant_id, kind, code) DO UPDATE SET name = EXCLUDED.name, name_en = EXCLUDED.name_en;

-- 中类（level=2，code 4 位）—— 用子查询拿父 id
INSERT INTO thingmodel_categories (kind, code, level, parent_id, name, is_enabled, tenant_id, created_at)
SELECT 'SYSTEM', v.code, 2, p.id, v.name, true, 0, NOW()
FROM thingmodel_categories p, (VALUES
  ('3001', '冷热源系统'),
  ('3002', '空调系统'),
  -- ...
) AS v(code, name)
WHERE p.kind = 'SYSTEM' AND p.code = LEFT(v.code, 2) AND p.tenant_id = 0
ON CONFLICT (tenant_id, kind, code) DO UPDATE SET name = EXCLUDED.name;

-- 小类（level=3，code 6 位）—— 父 code 是前 4 位
-- 细类（level=4，code 8 位）—— 父 code 是前 6 位
-- 略
```

→ v2 的 SQL 比 v1 简洁——无需 path 拼接、无需 level 参与唯一键。

## 5. 分阶段实施计划

| 阶段 | 任务 | 产出 | 工时估算（v2） | v1 估算 | 节省 |
|------|------|------|---------------|---------|------|
| **P0 后端骨架** | proto 两层 + ent schema + repo + 基础 CRUD service + wire + server + `make api/ent/wire` | 分类可 CRUD | **1.5~2 人日** | 2~3 人日 | ~30% |
| **P1 校验** | 8 项校验（V1–V12）+ 单元测试 | 校验可用、测试全绿 | **1 人日** | 1.5~2 人日 | ~40% |
| **P2 前端** | hooks + Tabs + 单 ProTable + Drawer + 路由 + i18n + 菜单/权限码 | 管理界面可用 | **1.5~2 人日** | 2~3 人日 | ~30% |
| **P3 种子数据** | 解析器 + 清单 embed + 600~750 节点 upsert + 验证 | 出厂默认数据齐全 | **1~1.5 人日** | 1.5~2 人日 | ~25% |
| **P4 联调验收** | 端到端联调 + 数据正确性抽查 + 性能验证 | 达验收标准 | **0.5~1 人日** | 1 人日 | ~30% |

## 7. 数据治理点

| 问题 | 处理 |
|------|------|
| **SPACE 仅 1 大类，要不要把"中类"提升为"大类"？** | **不调整**。坚持"清单是事实"。前端在 SPACE Tab 下用户从大类「建筑空间」开始展开即可。 |
| **v1 中"小类 30010100 与细类 30010100 同 code 冲突"问题** | **v2 已自然消除**——小类 code 是 `300101`（6 位），细类 code 是 `30010100`（8 位），长度不同。 |
| **大类编码段是否会与 unit 等其它 thingmodel 字典冲突？** | 不冲突。`thingmodel_categories` 与 `thingmodel_units` 是独立表，code 命名空间互相隔离。 |
| **未来如何引入新分类（如"文档分类"）** | **追加 `CategoryKind` 枚举值**（如 `DOCUMENT`），按需要可不再受"4 层 + 数字 code"约束——通过 Service 层的 KindValidator 注册器区分，详见 [07 章](./07-扩展性设计.md)。 |
| **运行时用户级批量导入** | 本期不做。种子完成出厂默认；用户在 UI 上单条新建/编辑即可满足"私有分类扩展"。 |

## 8. 风险与遗留

| 风险/遗留 | 说明 | 应对 |
|-----------|------|------|
| `reference_count` 无实际维护方 | 本期字段预留恒 0 | 待 `thing_property` / `device_instance` 等引用方落地时，由其 Create/Update/Delete 维护 |
| 不允许 Move（改 parent_id） | 本期 schema/Service 锁定 immutable | 国标分类极少调整结构；如未来需求出现，单独评估实现 |
| 清单更新需同步种子 | 清单 md 文件变更后需重新构建 | embed 已保证；种子运行命令化（`gow seed thingmodel-category`） |
| 极少数细类清单已被废弃 | 国标改版可能少量细类失效 | 通过 `is_enabled=false` 标记，不删除原数据 |

## 9. 未来演进（非本期）

1. **物模型属性引用**：`thing_property.category_id` → 本模块（细类 level=4）。
2. **设备/空间/系统实例**：`*_instance.category_id` 指对应 kind 的细类。
3. **跨 kind 关联**：建立 SYSTEM ↔ FACILITY 的弱关联表（如"暖通空调系统(30) 由 暖通空调设备(20) 组成"），用于 BOM 视图。
4. **分类版本化**：当 GB/T 51269 改版时，引入 `version` 字段或独立"清单版本"表。
5. **批量导入/导出**：UI 提供 Excel 导入（按 4 层格式），方便用户在系统种子之上扩展私有分类。
6. **国际化扩展**：若需多语种分类名，届时再拆 `category_i18n` 子表。
7. **接入文档/证书/表单分类**：通过 kind 枚举追加 + KindValidator 注册器实现，详见 [07 章](./07-扩展性设计.md)。
