# 01 · 架构决策（v2.0）

## 1. 背景

物模型领域已落地"单位管理"作为第一块基石。下一步给物模型提供**分类挂载锚点**：

- 物模型属性（`thing_property`，未来）需归属一个具体分类。
- 设备实例 / 空间实例 / 系统实例需归属一个**细类**（level=4）。

输入是项目下三份国标分类清单：

| 清单 | 编码段（大类） | 大类数 | 总细类数 | 标准依据 |
|------|--------|--------|----------|----------|
| `智能系统分类清单.md` | **30–36** | 7 | ≈170 | GB/T 51269-2017、DB31/T 1320-2021、T/BCAT 0002-2022 |
| `空间分类清单.md` | **10** | 1（下含 5 中类） | ≈100 | GB/T 51269-2017、T/CECS 1196-2022 |
| `设备设施分类清单.md` | **20–26** | 7 | ≈210 | GB/T 51269-2017、DB31/T 1320-2021、T/CCIAT 0040-2021 |

三套清单结构完全同构：**大类(2 位编码) → 中类(4 位) → 小类(6 位) → 细类(8 位)** 共 4 层。

## 2. 核心决策一：模块化，而非插件化

**结论同 [单位管理 01 章](../单位管理/01-架构决策.md) 第 2 节**：项目从后端到前端、从构建到运行，全程是**编译期模块化**，无插件注册表/动态加载器/运行时扩展点。本功能继续以"领域子能力"形态落入 `thingmodel` 域。

## 3. 核心决策二：单表 + kind 枚举承载多业务域

### 3.1 决策

> **使用一张 `thingmodel_categories` 表，加 `kind` 枚举字段区分多业务域**——本期 kind 包含 `SYSTEM / SPACE / FACILITY` 三值，未来扩展到 `DOCUMENT / CERTIFICATE / FORM` 等只需追加枚举值。**不**引入额外的 `scope` 字段，**不**为每套清单建独立表。

### 3.2 为什么 kind 已经足够（v2 vs v1 对比）

v1.0 设计中曾提议引入 `scope` 字段把"分类管理"通用化（`scope=thingmodel/document/...`），但回头看：**`kind` 字段本身就承担了业务域区分的语义**。两层抽象（scope + kind）的合理性需要满足"同 scope 下有多 kind 互相联系"——但实际：

- 物模型场景：SYSTEM/SPACE/FACILITY 三 kind 互不引用，各自独立 4 层结构。
- 文档场景：未来即便加上 DOCUMENT，它和 SYSTEM 也不会产生跨 kind 引用。

→ **scope 抽象在本项目找不到正当使用场景**，YAGNI。直接扩展 kind 枚举更简洁、零额外维护成本。

### 3.3 候选方案最终对比

| 方案 | 表数 | 字段维度 | 评价 |
|------|------|----------|------|
| A. **单表 + kind 枚举**（采纳） | 1 | kind 即业务域 | ✅ 最简洁，未来扩展只需追加枚举值 |
| B. 单表 + scope + kind（v1.0 旧方案） | 1 | 双维度 | ❌ scope 找不到第二个 kind 集合，过度设计 |
| C. 三表 / N 表 | N | 业务域=表 | ❌ 字段 100% 重复，前后端代码三份 |

> kind 命名沿用 v1.0 的 **`SYSTEM` / `SPACE` / `FACILITY`** 三值，对齐"智能系统/空间/设备设施"。

### 3.4 kind 枚举的扩展路径（未来）

```protobuf
enum CategoryKind {
  CATEGORY_KIND_UNSPECIFIED = 0;

  // ===== 本期：物模型三 kind =====
  SYSTEM   = 1;
  SPACE    = 2;
  FACILITY = 3;

  // ===== 未来：业务方接入时按需追加 =====
  // DOCUMENT    = 10;
  // CERTIFICATE = 11;
  // FORM        = 12;
  // ARTICLE     = 13;
  // ...
}
```

未来扩展不同业务的具体约束（层级深度、code 格式）由 **KindValidator 注册器**承担，详见 [07 章](./07-扩展性设计.md)。

## 4. 核心决策三：固定 4 层 + 变长 code（按 level 严格分级）

### 4.1 决策

> **层级固定 4 层**（大类→中类→小类→细类），用 `level: 1/2/3/4` 字段标记。
> **code 长度按 level 严格分级**：
> - level=1（大类）：**2 位**，如 `30`、`10`、`20`
> - level=2（中类）：**4 位**，如 `3001`、`1001`
> - level=3（小类）：**6 位**，如 `300101`
> - level=4（细类）：**8 位**，如 `30010100`、`30010101`
>
> **绝不补 0 凑长度**——大类就是 `30`，不写成 `30000000`。

### 4.2 为什么这样比 v1.0 的"全部 8 位"更好

| 维度 | v1.0 全 8 位 | v2.0 变长 code |
|------|-------------|---------------|
| 与清单对齐 | 大/中/小类需补 0 派生 | ✅ 大类标题就是 `## 30`、清单 code 列就是 8 位细类 |
| 唯一键 | 需 `(tenant, kind, code, level)` 4 元组，因为小类 `30010100` 与首细类 `30010100` 冲突 | ✅ **`(tenant, kind, code)` 3 元组就够**——code 长度不同自然不冲突 |
| 前缀查询 | `code LIKE '30%'` 取暖通系统 | ✅ 同样可行，且更精确：`code = '30' OR code LIKE '30__' OR code LIKE '30____' ...` |
| 父子关系推导 | 需补 0 后比对前缀 | ✅ **`parent.code = substr(self.code, 1, self.level*2-2)`**——直接截断 |
| 用户输入心智 | "为啥大类要补这么多 0？" | ✅ "大类就 2 位，中类 4 位"——所见即所得 |
| 视觉排序 | 全 8 位字符串排序 | ✅ 因为 2/4/6/8 位 code 仍是数字字符串前缀比较，字典序 = 层级序 |

### 4.3 排序友好性证明

按 code **字符串升序**排序的结果（取智能系统 SYSTEM kind）：

```
30                ← 大类「暖通空调系统」
3001              ← 中类「冷热源系统」
300101            ← 小类「制冷系统」
30010100          ← 细类「压缩式制冷系统」
30010101          ← 细类「吸收式制冷系统」
300102            ← 小类「供热系统」
30010200          ← 细类「集中供热系统」
30010201          ← 细类「独立供热系统」
3002              ← 中类「空调系统」
300201            ← 小类「全空气系统」
...
31                ← 大类「给排水系统」
3101              ← ...
```

→ **按字符串自然升序排列，恰好是深度优先遍历的顺序**。前端不需要任何树结构，单一 ProTable 按 code asc 排即层级清晰。

### 4.4 "不存在结尾为字符 0 的编码"——清单观察的进一步确认

仔细看清单，**所有出现在表格中的 code（细类）都是 8 位**，且**末两位**用作"细类序号"。但用户原始诉求"不存在结尾为字符 0 的 code"，意思是：**大/中/小类的 code 不应该用'补 0'的方式表达**——这恰好是 v2 决策的核心。

> ⚠️ **重要数据观察**：清单中**首个细类的 code 末两位通常是 `00`**（如 `30010100` 是「压缩式制冷系统」），这是国标编码规则——首个细类继承小类编码后补 `00`。
> 但是这里"末两位 00"是**细类自身的编号**（00=第一个、01=第二个、02=第三个...），**不是"补 0 凑位数"**，符合 v2 原则。
> 所以"不存在结尾为字符 0 的 code"这一原则的准确解读是：**不在 code 末尾补 0 来凑长度**，而清单中 `30010100`、`30010200` 这种细类 code 末两位 `00` 是合法的（语义上是该小类下的"第一个细类"序号）。

### 4.5 父子关系如何推导

```go
// 由子 code 求父 code：直接截断到父级长度
func parentCode(selfCode string, selfLevel uint8) string {
    if selfLevel <= 1 {
        return ""
    }
    parentLen := (selfLevel - 1) * 2  // level=4 → 父长度 6；level=3 → 父长度 4
    return selfCode[:parentLen]
}

// 由 code 推导 level
func levelOf(code string) uint8 {
    return uint8(len(code) / 2)  // len=2 → level=1；len=8 → level=4
}
```

→ level 字段甚至**可以由 code 长度推导**（冗余存储是为了索引和过滤性能）。

## 5. 核心决策四：不存 `is_leaf` 字段

### 5.1 决策

> **删除 v1.0 的 `is_leaf` 字段**。叶子判定直接由 `level == 4` 推导。

### 5.2 理由

- 本期约束："只有 level=4 的细类才是叶子节点"——这条规则**固定不变**。
- v1.0 的 `is_leaf` 字段是为了避免"扫子节点判 leaf"而做的冗余存储；但既然规则是 `level==4`，那么 `is_leaf` 就是 `level` 的一个布尔投影，**纯冗余**。
- 维护冗余字段需要额外的写一致性逻辑（新建子节点时把父 `is_leaf` 置 false），代码复杂度增加而无任何收益。

### 5.3 取舍：那"用户能不能给 level=3 小类直接挂属性？"

不能。这就是规则。规则用 `level == 4` 在 Service 层一行代码判断即可，比维护 `is_leaf` 字段简单得多。

## 7. 设计原则

1. **镜像 unit，不发明轮子**：表结构、proto 骨架、种子策略全部参照 [单位管理](../单位管理/)。
2. **国标约束硬化**：4 层 + 变长 code + kind 三元约束在 Schema、Service、UI 三处显式表达。
3. **结构化优于自由文本**：level、kind 都是 enum/小整数字段，不用魔法字符串。
4. **极简即正义**：能用 `level==4` 推导就不存 `is_leaf`；能用 code 排序就不用 Tree 组件；能用 kind 区分就不引入 scope。
5. **种子数据即契约**：3 套约 600 节点由种子从 markdown 清单解析生成，版本化管理。
6. **租户隔离**：系统预置分类 `tenant_id = 0`（全局），租户可扩展私有分类。

## 8. 非目标（本期不做）

- 物模型属性（Property）/事件/服务的定义与维护。
- 任何 `*_instance`（device/space/system_instance）实例化模块。
- 分类的版本管理（GB/T 51269 改版分支）。
- 跨 kind 的关联映射。
- 第 5 层及以下的自定义层级。
- 节点 Move（修改 parent_id）——分类一旦确立结构稳定。
- 分类的批量导入（仅做种子；用户级批量导入留待下期）。
