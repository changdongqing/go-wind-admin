# 14 · 特征种子数据与实施计划

> 本章从《物模型特征清单》提炼四类特征的种子数据规划，给出编码规则、分阶段任务、验收标准与数据治理点。

## 2. 编码规则（对齐清单 §9）

### 2.1 属性编码（沿用清单）

```
P-[类别]-[序号]
```

| 类别代码 | 含义 | 示例 |
|----------|------|------|
| RUN | 通用运行状态 | P-RUN-0001 |
| RAT | 通用额定参数 | P-RAT-0007 |
| STA | 通用统计参数 | P-STA-0001 |
| ENV | 通用环境参数 | P-ENV-0001 |
| HVAC | 暖通空调专属 | P-HVAC-0001 |
| WP/EL/FP/ELV/SM | 给排水/电气/消防/电梯/智能化 | P-WP-0001 |

额定特征编码：`RAT-[设备大类]-[设备小类]-[序号]`，如 `RAT-HVAC-CH-001`。

### 2.2 事件编码（沿用清单）

```
E-[类别]-[序号]    例：E-GEN-0001 / E-HVAC-0001
```

GEN=通用，其余类别代码同属性。

### 2.3 服务编码（沿用清单）

```
S-[类别]-[序号]    例：S-GEN-0001 / S-HVAC-0001
```

### 2.4 关系编码（本期新增）

```
R-[设备大类]-[序号]    例：R-HVAC-0001
```

- 大类代码与属性同（HVAC/WP/EL/FP/ELV/SM/AX）。
- 跨设备大类的关系用 `R-GEN-[序号]`。

## 3. 种子数据生成策略

### 3.1 推荐方案：Go 种子程序

新增 `backend/app/admin/service/internal/data/seed/feature_seed_data.go` + `feature_seed.go`（与单位种子同目录、同模式）：

```go
// Package seed 提供物模型特征种子数据（幂等 upsert）。
// 设计依据：docs/thingmodel/sheji/14-特征种子数据与实施计划.md
// 数据来源：docs/thingmodel/物模型特征清单.md

type SeedFeature struct {
    FeatureType     thingmodelV1.FeatureType
    Code            string
    Identifier      string
    Name            string
    NameEn          string
    Description     string
    ApplicableScope string   // 适用设备，如 "冷机/锅炉" 或 "通用"
    SortOrder       uint32
    Spec            map[string]any   // spec（map 字面量保持可读；upsert 时转为 proto FeatureSpec oneof 落库）
}

// 缩写
const (
    ftProperty = thingmodelV1.FeatureType_PROPERTY
    ftEvent    = thingmodelV1.FeatureType_EVENT
    ftService  = thingmodelV1.FeatureType_SERVICE
    ftRelation = thingmodelV1.FeatureType_RELATION
)
```

> relation 的 source/target 用 `identifier` 而非 `id`（种子录入时 id 未知，按 identifier 解析）——种子程序 upsert 后需回填 id（见 §3.3）。

### 3.2 幂等策略

与单位种子一致：
- 按 `(tenant_id=0, code)` upsert。
- 仅维护 `tenant_id=0` 系统预置数据；租户自建（tenant_id>0）永不被覆盖。
- upsert 前先跑 `validateFeatureSpec`，保证种子 spec 合法。

### 3.3 关系种子的 id 回填

relation 的 source/target 用 identifier 引用，upsert 后需解析为 id：

```go
func SeedThingmodelFeatures(ctx context.Context, client *ent.Client) error {
    // 第一遍：upsert 所有非 relation 特征（property/event/service）
    for _, f := range FeatureSeedData {
        if f.FeatureType == ftRelation { continue }
        upsertFeature(ctx, client, f)   // 按 code upsert，回填 id
    }
    // 第二遍：upsert relation 特征，解析 source/target.identifier → id
    for _, f := range FeatureSeedData {
        if f.FeatureType != ftRelation { continue }
        resolveRelationRefs(ctx, client, f)  // identifier → id
        upsertFeature(ctx, client, f)
    }
    return nil
}
```

> **触发时机**：与单位种子一致，做成**显式命令**（`gow seed` 或启动 hook），非每次启动自动跑。

### 3.4 单位引用处理

属性种子的 `spec.unit.unitCode` 需在 upsert 时解析为 `unitId`（查 `thingmodel_unit`）：

```go
// upsertFeature 内，property 类型时
if f.FeatureType == ftProperty {
    if unitCode, ok := f.Spec["unit"].(map[string]any)["unitCode"].(string); ok {
        unitID := lookupUnitIDByCode(ctx, client, unitCode)  // 查 unit.id
        f.Spec["unit"].(map[string]any)["unitId"] = unitID
    }
}
```

> 依赖单位种子已先执行（unit 表有数据）。种子执行顺序：**单位种子 → 特征种子**。

## 4. 分阶段实施计划

| 阶段 | 任务 | 产出 | 工时估算 |
|------|------|------|----------|
| **P0 后端骨架** | proto 两层 + ent schema + repo + 基础 CRUD service + wire + server 注册 + `make api/ent/wire` | 特征可 CRUD，Swagger 可调 | 2~3 人日 |
| **P1 spec 校验器** | feature_validator.go 四类校验 + ValidateSpec RPC + 特化列同步 + 单元测试 | 校验可用、测试全绿 | 2 人日 |
| **P2 单位引用打通** | unit_repo.IncReferenceCount + feature_service 引用维护 + 关系完整性校验 | 挂钩点兑现 | 1 人日 |
| **P3 前端** | hooks + Splitter 页面 + 类型树 + 动态表单（四类）+ 路由 + i18n + 权限码 | 管理界面可用 | 3~4 人日 |
| **P4 种子数据** | ~330 特征种子程序 + 关系 id 回填 + 单位引用解析 + 迁移验证 | 出厂默认数据齐全 | 2~3 人日 |
| **P5 联调验收** | 端到端联调 + 校验规则抽查 + 性能验证 | 达验收标准 | 1 人日 |

**合计约 11~14 人日**。P0/P1 后端先行，P3 前端可与 P1 部分并行（hooks 可在 P0 接口稳定后先行），P4 依赖 P0 表结构 + 单位种子。

### 4.1 关键路径与并行

```
P0(后端骨架) ──┬─▶ P1(spec校验) ──▶ P2(单位引用) ──┐
               └─▶ P3(前端) ────────────────────────┼─▶ P5(联调验收)
P4(种子数据) ─────────────────────────────────────────┘
```

- P0 接口（proto）一稳定，P3 hooks 即可开工。
- P4 依赖 P0 表结构 + 单位种子（已落地）。
- P2 依赖 P1 校验器（引用维护需 spec 合法）。

## 5. 验收标准

### 5.1 功能验收

- [ ] 统一特征表落地，四类特征 CRUD 正常，Swagger 可见全部 9 个 REST 接口。
- [ ] 特征类型切换（左侧树）→ 右侧列表列动态变化。
- [ ] 四类动态表单按 feature_type 正确渲染（property 含数据类型/单位/struct 递归；event 含级别/输出参数；service 含调用模式/输入输出；relation 含 source/target）。
- [ ] 校验规则全部生效（V1~V17）：dataType 必填、enum 项非空、struct 子字段非空、min≤max、关系目标存在等。
- [ ] `code`/`identifier` 租户内唯一；特化列与 spec 一致（F4/F17）。
- [ ] 权限码控制：无权限不显示按钮/无法访问路由。

### 5.2 单位引用挂钩点（打通）

- [ ] 创建 property（含 unit 引用）→ `unit.reference_count` +1。
- [ ] 删除 property → `unit.reference_count` -1。
- [ ] 删除被 property 引用的 unit → 返回 `UNIT_IN_USE_CANNOT_DELETE`（单位管理 C6 约束生效）。
- [ ] property 表单单位选择器调 `useListUnitsByCategory` 正常工作。

### 5.3 关系完整性

- [ ] 创建 relation（source/target 指向存在的 feature）成功。
- [ ] source.kind=feature 但目标不存在 → `FEATURE_RELATION_TARGET_NOT_FOUND`。
- [ ] 删除被 relation 引用的 feature → `FEATURE_IN_USE_CANNOT_DELETE`。

### 5.4 种子数据

- [ ] ~330 特征种子成功入库（`tenant_id=0`），幂等可重复执行。
- [ ] relation 种子的 source/target 正确解析为 id。
- [ ] property 种子的 unit 引用正确解析为 unitId。
- [ ] 种子数据与《特征清单》编码、标识符、参数一致。

### 5.5 性能

- [ ] `ListByType`（某类型全量特征）< 50ms（有索引）。
- [ ] `ValidateSpec` < 10ms。
- [ ] 种子 upsert 330 条 < 5s。

### 5.6 代码质量

- [ ] `make lint` 通过；无手改 `gen/`、`ent/`、`*_wire_gen.go`、`api/generated/`。
- [ ] 中英双语注释；Conventional Commits。

## 6. 数据治理点（落地时需决策）

| 问题 | 处理 |
|------|------|
| 额定参数（铭牌）是属性还是独立类型 | 归入 PROPERTY，`category=rated` + `isRated=true`。铭牌本质是只读属性，无需独立类型 |
| "适用设备"列与未来 product/device 的关系 | 本期 `applicable_scope` 是自由文本（如"冷机/锅炉"）；未来可改为引用设备分类字典 |
| relation 的反向关系维护 | 本期不自动维护（feeds/suppliedBy 等成对关系需手动建两条）；未来可加"反向创建"快捷操作 |
| struct 嵌套深度 | 前端建议限制 ≤ 3 层；后端不硬限制（DB/JSON 支持） |
| spec JSON 查询的 DB 兼容性 | `ReferencedByRelation` 用 PostgreSQL JSON 查询；MySQL 需改 `JSON_EXTRACT`。按 `data.yaml` 确认 |
| 多语言（unit 的 name_en 已有） | feature 复用 name/name_en 双语；未来需多语种再拆 i18n 子表 |

## 7. 风险与遗留

| 风险/遗留 | 说明 | 应对 |
|-----------|------|------|
| spec JSON 无 DB 级约束 | 复杂结构靠应用层校验器 | 校验器单测全覆盖；种子入库前强制校验 |
| 关系引用一致性 | source/target 用 JSON 存，无 DB 外键 | 应用层校验 + 删除前置检查（F11） |
| 特化列与 spec 双写一致性 | 同一信息存两处（列 + spec） | `syncSpecializedColumns` 写入时同步；单测验证 |
| 种子数据量大（330+） | 录入与维护成本 | 分设备大类批次录入；种子源码版本化 |
| relation 的 external 引用未校验 | 指向未建的 device/space 表 | 本期弱校验；待 product/device 落地后补强 |
| 单位引用计数最终一致性 | 引用维护失败仅记日志不回滚 | 定期对账任务（统计 property 引用 → 校正 reference_count） |

## 8. 未来演进（非本期）

1. **产品物模型模板**：将特征组装成产品模板（`product`），设备实例继承。届时特征引入版本化（被产品引用后不可随意改 spec）。
2. **设备实例 + 数据上报**：设备绑定产品物模型，上报数据按 property spec 校验 + 单位换算（打通单位管理的设备归一化运行时）。
3. **关系图谱可视化**：基于 relation 数据渲染知识图谱（D3/Cytoscape），支持设备拓扑、属性派生链可视化。
4. **特征继承与覆盖**：产品模板引用全局特征库的特征后，可覆盖部分 spec（如修改 min/max），形成"实例化特征"。
5. **特征的版本与发布**：特征变更走版本管理（draft/published），产品引用指定版本。
