# 01 · 架构决策

> 本章回答两个问题：
> 1. "模型管理"为何不作为独立菜单存在？为什么拆成"分类默认模型 + 产品管理"两个入口？
> 2. 数据层为什么采用"双表拆分"（默认模型一表、产品两表）而非"统一模型表 + model_kind"或"JSON 大对象"？

---

## 1. 形态判定：双入口 而非 独立菜单

### 1.1 用户原始诉求

> "本身其实并不存在一个模型管理的菜单，模型的配置隐藏在两个功能中"

这一表述蕴含的核心洞察：**"模型"是一个跨越多个治理场景的逻辑实体，不是一个独立可治理的资源**。强行建一个"模型管理"菜单会带来三个问题：

1. **语义重复**：分类、产品本就承载"建模"语义；再加"模型"菜单就出现"分类→默认模型→产品"三段式入口而模型本身没有独立增删改的入口。
2. **用户心智断裂**：用户想给"冷水机组"配几个属性 — 他从分类管理进入还是从模型管理找？答案不应该让用户来记。
3. **数据冗余链路**：抽象 `model` 实体后，"分类默认模型"和"产品模型"会都变成 model 的两种 instance，但它们的生命周期、引用计数、覆写语义完全不同（默认模型是"模板"、产品模型是"实例"），强行抽象徒增复杂度。

### 1.2 双入口的语义对齐

| 入口 | 语义身份 | 业务时态 | 入口位置 |
|------|---------|---------|---------|
| 分类默认模型 | **类型模板**（class template） | 设计期：建库者维护"这一类设备应有的特征" | 分类管理 level=4 行内 Drawer |
| 产品模型 | **可实例化原型**（concrete prototype） | 实施期：用户/集成商定义"这台格力 KFR-72 长什么样" | 物模型 / 产品管理（新菜单） |

> 这种二分法对齐工业本体方法论（DOLCE / SUMO / BFO）：**Class → SubClass/NamedIndividual** 的链路，对应"分类 → 产品"。
> 与阿里云 IoT 的"品类（catalog）→ 产品（product）"、华为云 IoTDA 的"产品模型（product model）→ 产品（product）"心智一致。

### 1.3 候选方案对照（被否决）

| 方案 | 否决理由 |
|------|---------|
| **A. 独立"模型管理"菜单**，把分类默认模型和产品模型都放进去 | 违背用户表述；语义混杂；引入 model 抽象层徒增复杂度 |
| **B. 不区分默认模型，产品直接选特征建模** | 失去"分类知识沉淀"价值；每个新产品都要从零搭建；不符合分类管理"分类知识库"的定位 |
| **C. 默认模型放在分类管理一级菜单的独立子页面** | 入口分散，用户在分类列表上看不到"是否已配模型"的状态信号；Drawer 形态更聚合 |

### 1.4 决策

**采用双入口形态**：分类管理增量"行内 Drawer 配置默认模型" + 新建"产品管理"菜单。

## 2. 表结构策略：双表拆分

### 2.1 备选方案对照

| 方案 | 数据模型 | 优点 | 缺点 |
|------|---------|------|------|
| **A. 双表拆分（采纳）** | `category_default_features` 关联表<br>`products` + `product_features` 两表 | 语义清晰；CRUD 直接；reference_count 维护点单一；与单位管理/特征管理同构 | 表数量从 0 变 3 |
| B. 统一模型表 + model_kind | `models`(model_kind=CATEGORY_DEFAULT/PRODUCT) + `model_features` | 表更少 | model_kind=CATEGORY_DEFAULT 与 category 1:1 是冗余中间表；二者业务规则差异巨大（覆写白名单不同、source 字段只对 PRODUCT 有意义、生命周期/级联策略不同），强行同表反而要写"if model_kind==X then..."的胶水代码 |
| C. 都用 JSON 数组 | category 表追加 `default_model_spec JSON`，products 表追加 `features JSON` | 查询最快、表最少 | reference_count 难维护（特征/单位被引用关系藏在 JSON 里）；不能按特征反向查询"哪些产品/分类用到了 XX 特征"；改单个 feature 需读改写整个 JSON，并发冲突频繁 |

### 2.2 双表拆分的关键收益

1. **`reference_count` 真正可维护**：每次新增/删除关联行就 ±1，无需扫 JSON。
2. **反向索引天然可建**：`INDEX (feature_id)` 直接答"哪些产品/分类引用了我"。
3. **覆写白名单各管各的**：default_features 与 product_features 校验器代码独立，无 if-else 分支。
4. **CASCADE/RESTRICT 表达力**：`product_features` 走 `ON DELETE CASCADE`（删产品连带删特征），`default_features.category_id` 走 `RESTRICT`（防误删分类），DB 层即可表达。
5. **同构其它模块**：与"单位 unit_category↔unit"、"特征 thing_features"的 ent.Schema 风格一致，认知零迁移。

### 2.3 三表的职责切分

```
┌───────────────────────────────┐  ┌─────────────────────────┐  ┌──────────────────────────────┐
│ category_default_features     │  │ products                │  │ product_features             │
│                               │  │                         │  │                              │
│ 职责：分类→特征的多对多关联     │  │ 职责：产品主元数据       │  │ 职责：产品下特征条目          │
│ (含 override_spec 轻量覆写)    │  │ (code/category_id/...)  │  │ (含 feature_snapshot 快照)    │
│                               │  │                         │  │                              │
│ 生命周期：随分类               │  │ 生命周期：独立           │  │ 生命周期：CASCADE 跟随产品    │
│ 唯一性：(category_id, feat_id) │  │ 唯一性：(code) +         │  │ 唯一性：(product_id, code) +  │
│                               │  │   (category_id, name)   │  │   (product_id, identifier)   │
└───────────────────────────────┘  └─────────────────────────┘  └──────────────────────────────┘
```

## 3. 继承语义：值复制（深拷贝）

### 3.1 备选方案对照

| 方案 | 行为 | 否决/采纳理由 |
|------|------|---------------|
| **A. 值复制（采纳）** | 产品创建时把 default_features 的 spec 深拷贝到 product_features.feature_snapshot；默认模型后续变更不影响已建产品 | 产品自洽、零运行期 join、语义直观；与阿里云"从品类拷贝"一致 |
| B. 引用继承（link） | product_features 只存 ref_feature_id + override；effective_spec = default_features.spec + override + feature.spec 三层合并 | 一致性强但理解负担大；默认模型一改，所有产品"暗中"被影响；难以审计 |
| C. Snapshot + 主动升级 | 值复制时记录 source_version；默认模型升级时 UI 提示"有可用升级，是否拉取" | 表达力最强但需要"模型版本"基础设施，本期未做 |

### 3.2 为什么不是 B

引用继承的代价是**复杂性外溢到全栈**：
- DB 层无法在一次查询里返回完整 effective_spec；
- Service 层每次返回 product_feature 都要做三层 merge；
- 前端无法在不调用后端的情况下编辑（merge 算法在后端）；
- 一旦默认模型/全局特征改了 dataType（不允许但难拦），产品就破了。

值复制以"少量存储冗余"换取"全栈简单 + 产品自洽"，是 IoT 平台主流选择。

### 3.3 升级链路（本期不实现）

预留方案 C 的接口形态：

```
product_features 表追加可选字段（本期 NULL）：
  source_version   uint32   -- 拷贝时 default_features.updated_at 的版本号
  upgrade_available bool    -- 由后台任务定期对比 set
```

未来 UI 在 product_features 列表上展示"⬆ 可升级"标记，用户点击后弹出 diff 与"应用升级"按钮 — **本期不做，但表结构兼容**。

## 4. 覆写白名单：为什么不让改 dataType

### 4.1 白名单字段

```
constraints   ── {min, max, step}
unit          ── {unitId, unitCode, unitSymbol}
defaultValue
displayName   ── 在产品/分类局部展示别名
description
required      ── 仅 service 输入参数子项
sort_order
is_enabled
```

### 4.2 黑名单字段（强校验拒绝）

```
feature_type      ── 改这个等于"换特征"
dataType          ── 同上
accessMode        ── R/RW 决定上下行通道，不可乱改
enumItems 结构    ── 删/加枚举项会破坏设备协议
structFields 结构 ── 同上
arraySpec.element ── 同上
identifier        ── 设备协议映射标识符，跨产品复用，不可改
code              ── 程序标识，immutable
```

### 4.3 原则

> **结构性改变 = 另起一个特征**；只有"边界/标注"才走覆写。

试图在产品上把"温度 double, -20~60"覆写成"温度 int, 0~50" — 工程上等同于换了一个属性，应该新建 source=LOCAL 的特征。

这条原则上溯到本体设计：**Properties 的 range 和 domain 是 Property 的同一性标识，不可在子类层面替换**；这与覆写白名单逐字对应。

## 5. 范围与非目标

### 本期做

- 分类默认模型 CRUD（仅 level=4）
- 产品 CRUD + 产品下特征 CRUD（三类来源 DEFAULT/GLOBAL/LOCAL）
- "从默认模型批量拉取"批量操作 API
- override_spec 白名单校验
- reference_count 真正维护（终结单位管理的预留挂钩点）
- 菜单 + 权限码 + 默认角色绑定（`sys:platform_admin` / `sys:tenant_manager`）

### 本期不做（见 README §"非目标"）

- 产品模型版本号 / 升级广播
- 设备实例 / 数据上报
- 批量 Excel 导入
- 跨租户克隆
- 物模型 OTA 出码

---

## 6. 与现有 CLAUDE.md 规则的对齐

- ✅ **零侵入流水线**：proto → `make api` → `make ts`，三层架构 Service/Repo/Wire
- ✅ **菜单 + 权限码**：在 `backend/pkg/constants/default_data.go` 的 `DefaultMenus` 追加 `物模型/产品管理` 一条；权限码自动派生为 `thingmodel:product:view/edit/create/delete`，绑定 `sys:platform_admin` + `sys:tenant_manager` 角色
- ✅ **proto JSON 安全**：`feature_snapshot` 和 `override_spec` 字段使用 `field.JSON(_, &thingmodelV1.FeatureSpec{})` 强类型目标 + `XxxSpecField` wrapper 类型（参考 CLAUDE.md "Step 13" 强制清单），杜绝 `cannot unmarshal ... into isXxx_Yyy` 问题
- ✅ **bilingual 注释**：所有 Ent Schema、proto message 均按"`// 中文 / English`"风格

## 7. 决策摘要

| # | 决策 | 选择 |
|---|------|------|
| 1 | 形态 | 双入口（分类管理 Drawer + 产品管理新菜单），无独立"模型管理"菜单 |
| 2 | 表结构 | 双表拆分（默认模型 1 关联表 + 产品 2 表） |
| 3 | 继承语义 | 值复制（深拷贝快照） |
| 4 | 拉取粒度 | 逐特征勾选，默认全选 |
| 5 | 默认模型层级 | 仅 level=4 细类 |
| 6 | 默认模型 kind | 所有 kind（含未来扩展） |
| 7 | 产品-分类 | 单细类归属 |
| 8 | 产品唯一性 | (tenant, code) + (tenant, category_id, name) 双唯一 |
| 9 | 制造商/型号 | optional 字段 |
| 10 | 覆写 | 白名单：constraints/unit/defaultValue/displayName/description/required/sort/enabled |
| 11 | 产品本地特征 | 允许（source=LOCAL，不写回全局） |
| 12 | source 枚举 | DEFAULT / GLOBAL / LOCAL（3 类，预留扩展空间） |
| 13 | 版本 | 本期不做，保留 `status: DRAFT/PUBLISHED` 字段 |
| 14 | Relation | 包含（4 类特征对等支持） |
