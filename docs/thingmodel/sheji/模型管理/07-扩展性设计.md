# 07 · 扩展性设计

> 本章列出本期**有意预留**的扩展挂钩点：什么能力延后做、表结构/接口如何兼容、未来落地时大致路径。
> 凡列入本章的扩展项，本期**不实现**，但保证未来落地**零迁移 / 仅追加字段**。

---

## 1. 扩展挂钩矩阵

| # | 扩展项 | 触发场景 | 预留方式 | 未来落地工作量 |
|---|--------|---------|---------|---------------|
| EX1 | 产品模型版本号 | 客户要求物模型"历史可追溯" | `products` 表追加 `version` / `published_at` 字段（本期 NULL） | S |
| EX2 | 默认模型升级广播 | 默认模型改后批量提醒已建产品 | `product_features` 追加 `source_version` / `upgrade_available` 字段 + 后台 diff 任务 | M |
| EX3 | 设备实例 device_instance | 真正"接设备进来" | `products.reference_count` 已预留；新建 `device_instances` 表 FK 产品 | L |
| EX4 | 跨租户产品克隆 | SaaS 场景租户互导模板 | RPC `CloneProductToTenant`；DB 层无须改 | M |
| EX5 | 批量导入 / 导出 | Excel/JSON 批量建产品 | RPC `ImportProducts` / `ExportProducts` + 文件存储复用现有 OSS | M |
| EX6 | "INHERITED" 来源类型 | 区分"未经覆写的拷贝" vs "已覆写的拷贝" | `source` 枚举追加 `INHERITED` 值（向前兼容） | S |
| EX7 | 产品分组 / 标签 | 同一分类下产品按品牌/系列分组 | 新建 `product_tags` 多对多关联表 | M |
| EX8 | 物模型 JSON 出码（设备协议） | OTA / 设备配网下发 | Service 增加 `ExportProductModel(productId) → JSON` RPC | M |
| EX9 | 产品级 simulator / 联调 | 上线前调试 | 独立 simulator 服务依赖产品 JSON 出码 | L |
| EX10 | 跨 kind 的混合默认模型 | 一个产品同时挂 FACILITY + SPACE | `product_categories` 多对多关联表替换 `products.category_id` 单列 | L（改 schema 较大） |
| EX11 | 产品继承（A 继承自 B） | 系列产品快速派生 | `products.parent_product_id` 自引用 + 拷贝逻辑 | M |
| EX12 | 关系（Relation）source/target 引用产品 | 产品间组合关系 | RelationSpec 内 `target.kind = "PRODUCT"` 已留扩展点 | S |
| EX13 | 默认模型作用于上层分类节点 | 大类承载"通用属性" | `category_default_features` schema 不变；放开 Service 校验 `level==4` | S |
| EX14 | 产品级单位偏好 | 同一温度属性产品 A 用 ℃、产品 B 用 ℉ | 已支持：override_spec.unit | 已落地 |
| EX15 | 自定义覆写白名单 | 高级用户希望放宽 | 抽 `OverridePolicy` 配置项，默认值 = 当前白名单 | M |

## 2. 几个最常被问到的扩展项的详细路径

### 2.1 EX1: 产品模型版本号

**未来字段**（本期 NULL）：

```go
field.Uint32("version").Default(0).Nillable().
    Comment("产品模型版本号 / Product model version"),
field.Time("published_at").Optional().Nillable().
    Comment("最近一次 Publish 时间 / Last published time"),
```

**Publish RPC 语义升级**：

```protobuf
message PublishProductRequest {
  uint32 id = 1;
  string version_label = 2;   // "v1.0" / "v1.1" 之类
  bool   bump_minor = 3;       // true=自增小版本号
}
```

**版本快照**：新建 `product_published_versions` 表存历史。本期不动。

### 2.2 EX2: 默认模型升级广播

**未来字段**（本期 NULL）：

```go
// product_features schema 追加
field.Uint32("source_version").Optional().Nillable().
    Comment("拷贝时默认条目的 updated_at 版本号 / Source default feature version"),
field.Bool("upgrade_available").Default(false).
    Comment("默认模型已升级 / Default model has been upgraded"),
```

**后台任务**：定时（每小时/手动触发）对比 `product_features` 的 `source_version` 与 `category_default_features.updated_at`；不一致则置 `upgrade_available=true`。

**UI 提示**：产品详情页 Tab 标签上挂红点；行内 `⬆ 可升级` 按钮 → 弹出 diff Drawer，选择"应用"则把新 snapshot 拉过来并清空 override（或保留按字段询问）。

### 2.3 EX6: INHERITED 来源类型

当前 `source` 枚举 = `DEFAULT / GLOBAL / LOCAL`。

未来增 `INHERITED`：含义 = "完全继承自默认模型，未经任何覆写"。
- 进 DB 时 = `DEFAULT` 但 `override_spec IS NULL`。
- 编辑 override 后自动从 `INHERITED` 升级为 `DEFAULT`。

> 本期不引入是因为状态机增加了**显式枚举值**，但实质能力可由 `(source=DEFAULT, override_spec IS NULL)` 反推得到，YAGNI。

### 2.4 EX10: 多分类混合归属（破坏性）

如果未来确实需要"一个产品挂多个分类"：

- 新建 `thingmodel_product_categories` 多对多表 `(product_id, category_id, role)`，`role ∈ {PRIMARY, SECONDARY}`。
- 数据迁移：把现有 `products.category_id` 写到 `thingmodel_product_categories` 作为 PRIMARY 一条。
- `products.category_id` 可保留作为"主分类快查列"或删除。
- "拉取默认模型"语义变化：先按 PRIMARY 拉，再询问是否合并 SECONDARY 默认模型（重复特征如何处理需要新规则）。

**这是本期决策"单细类归属"被故意收窄的代价**。如果业务确实需要，建议**先量化频次**再决定要不要破坏性升级。

### 2.5 EX8: 物模型 JSON 出码

未来需要把产品物模型导出为标准 JSON（如阿里云 TSL / 华为云 product model）下发给设备：

```protobuf
rpc ExportProductModel(ExportProductModelRequest) returns (ExportProductModelResponse);

message ExportProductModelRequest {
  uint32 product_id = 1;
  ExportFormat format = 2;  // ALIYUN_TSL / HUAWEI_IOTDA / GENERIC_JSON
}
message ExportProductModelResponse {
  bytes payload = 1;        // JSON 字节
  string content_type = 2;  // application/json
}
```

**实现**：service 层遍历 product_features，按 effective_spec 渲染到目标格式。**完全用读侧**，不动 schema。

## 3. 与本体（Ontology）的对接路径

本设计是工程化的"分类 → 产品 → 设备"链路。如果未来要走更严格的本体方法（OWL/RDF）：

| 工程层概念 | 本体层概念 | 对接方式 |
|------------|-----------|---------|
| Category | Class | 1:1 |
| Product | Class（subClassOf Category 的子类）/ NamedIndividual | 由场景定；建议 Class 派生路径 |
| Feature(PROPERTY) | DatatypeProperty | property 的 dataType → xsd:int/double/string |
| Feature(EVENT) | Class（subClassOf :Event）+ 关联 Property | 事件即 ABox 事件实例 |
| Feature(SERVICE) | OWL Operation / SHACL Function | 通过命名空间映射 |
| Feature(RELATION) | ObjectProperty | source/target → domain/range |
| product_feature.feature_snapshot | 在 TBox 中通过 rdfs:range / owl:hasValue 体现 | — |
| override_spec.constraints | xsd:minInclusive / xsd:maxInclusive / SHACL constraints | 直接映射 |

**导出 RDF/OWL** 的路径：新增 `ExportOntology(categoryId or productId, format=TURTLE/JSONLD/OWLXML)` RPC。这是 EX8 的本体变种，完全读侧。

## 4. 与现有模块的耦合解耦

本设计**有意做窄**的几个耦合点，未来都能放开：

| 现耦合 | 放开路径 |
|--------|---------|
| 默认模型仅 level=4 | Service 改一行校验；schema 不变（EX13） |
| 产品仅挂 FACILITY 是 UI 约束，DB 层无此限制 | 直接放开 UI（前端 picker 取消 kind 过滤） |
| override_spec 白名单 | 改 `FeatureOverrideSpec` proto 追加字段 + 校验器（EX15） |
| 产品-分类 1:1 | 见 EX10 |
| Relation source/target 仅可指特征 | RelationSpec.target.kind 已是枚举（"feature"/"external"），追加 "product" 值即可（EX12） |

## 5. 不打算扩展的项（明确否决）

| 不做的事 | 理由 |
|---------|------|
| 把 `model` 抽象为独立实体表 | 默认模型 vs 产品模型生命周期/语义差异大，[01 §2](./01-架构决策.md) 已论述 |
| 允许产品覆写 dataType 等结构字段 | 结构性改变 = 另起特征；[01 §4](./01-架构决策.md) 已论述 |
| 默认模型 link 继承（产品引用而非拷贝） | 强外溢复杂度；[01 §3.2](./01-架构决策.md) 已论述 |
| 在产品上做"运行时模型"（活模型） | 物模型层是设计期产物；运行时态属设备实例职责 |

## 6. 决策与扩展性一览

| 本期决策 | 是否能优雅放开 | 路径文档 |
|---------|---------------|---------|
| 单细类归属 | 能（破坏性） | §2.4 |
| 一类一模 | 能（追加 variant_id） | — |
| 值复制 | 能但代价大 | [01 §3](./01-架构决策.md) |
| 仅 level=4 配模型 | 能（改 Service） | §4 |
| 仅 FACILITY 创建产品（UI） | 能（前端放开） | §4 |
| 无版本 | 能（追加 version 列） | §2.1 |
| 覆写白名单 | 能（policy 配置） | EX15 |
| product_feature.source 三类 | 能（枚举追加 INHERITED） | §2.3 |

**整体扩展性评估**：除"多分类归属"是破坏性、"link 继承"是大改外，**其余 13 项扩展均为追加字段或新增 RPC，零迁移 + 兼容现存数据**。
