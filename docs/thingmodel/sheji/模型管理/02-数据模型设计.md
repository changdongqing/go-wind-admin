# 02 · 数据模型设计

> 本章定义模型管理的领域模型、ER 图，及三张表 `thingmodel_category_default_features` / `thingmodel_products` / `thingmodel_product_features` 的字段、索引、约束。
> 表结构风格严格对齐已有 `thingmodel_units`、`thing_features`、`thingmodel_categories`。

---

## 1. 领域模型


### 1.1 关键设计抉择

| 抉择点 | 方案 | 理由 |
|--------|------|------|
| 是否抽象 `model` 表 | **不抽象**，直接三表 | model 实体生命周期/字段语义差异大，抽象徒增复杂度。详见 [01 §2](./01-架构决策.md) |
| 默认模型 vs 产品模型 spec 存哪 | 默认模型：`override_spec` 稀疏覆写<br>产品模型：`feature_snapshot` **完整快照** + `override_spec` 稀疏覆写 | 默认模型语义是"分类要求"，复用全局特征 spec 即可；产品模型必须自洽，每行存完整快照 |
| feature_snapshot 是否需要 | **需要**（产品自洽核心） | 全局特征改 spec 后产品不受影响；离线读取一次即可拿到完整产品物模型 |
| product_features 是否抽冗余列 | **抽**（feature_type / data_type / access_mode / event_level / call_mode / relation_type） | 列表筛选高频、避免 JSON 索引、便于 SQL 直接 group by |
| identifier / code 是否产品内重新唯一 | **是**（产品内唯一） | LOCAL 特征可与全局同名；不在产品内强制唯一就允许 LOCAL 撞上 DEFAULT，混乱 |
| `category_id` 校验 level=4 | Service 层 | DB 层无法表达跨表条件约束，应用层兜底 |

## 2. 表结构

### 2.1 `thingmodel_category_default_features`（分类默认模型条目）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | uint32 | PK, auto_increment | mixin.AutoIncrementId |
| `category_id` | uint32 | NOT NULL, FK→thingmodel_categories.id, ON DELETE RESTRICT | 仅允许 level=4 节点（Service 校验） |
| `feature_id` | uint32 | NOT NULL, FK→thing_features.id, ON DELETE RESTRICT | 引用全局特征库 |
| `override_spec` | json | optional | 稀疏 JSON，仅记录覆写字段（白名单约束） |
| `display_name` | string | optional | 在该分类下的展示别名 |
| `is_enabled` | bool | default true | mixin.IsEnabled |
| `sort_order` | uint32 | default 0 | mixin.SortOrder |
| `tenant_id` | uint32 | default 0 | mixin.TenantID，0=系统全局 |
| `created_by/updated_by/deleted_by` | uint32 | optional | mixin.OperatorID |
| `created_at/updated_at/deleted_at` | timestamp | optional | mixin.TimeAt |

#### Ent Schema 骨架

```go
package schema

import (
    "entgo.io/ent"
    "entgo.io/ent/dialect/entsql"
    "entgo.io/ent/schema"
    "entgo.io/ent/schema/edge"
    "entgo.io/ent/schema/field"
    "entgo.io/ent/schema/index"

    "github.com/tx7do/go-crud/entgo/mixin"

    thingmodelV1 "go-wind-admin/api/gen/go/thingmodel/service/v1"
)

// CategoryDefaultFeature 分类默认模型条目（关联表）
type CategoryDefaultFeature struct{ ent.Schema }

func (CategoryDefaultFeature) Annotations() []schema.Annotation {
    return []schema.Annotation{
        entsql.Annotation{
            Table:     "thingmodel_category_default_features",
            Charset:   "utf8mb4",
            Collation: "utf8mb4_bin",
        },
        entsql.WithComments(true),
        schema.Comment("分类默认模型条目 / Category default model entry"),
    }
}

func (CategoryDefaultFeature) Fields() []ent.Field {
    return []ent.Field{
        field.Uint32("category_id").
            Comment("分类 ID（必须 level=4）/ Category id, must be level=4"),
        field.Uint32("feature_id").
            Comment("全局特征 ID / Global feature id"),

        // 稀疏覆写（白名单：constraints/unit/defaultValue/displayName/description/required）
        field.JSON("override_spec", &thingmodelV1.FeatureOverrideSpec{}).
            Optional().Nillable().
            Comment("稀疏覆写 / Sparse override spec"),

        field.String("display_name").Optional().Nillable().
            Comment("分类内展示别名 / Display alias within category"),
    }
}

func (CategoryDefaultFeature) Mixin() []ent.Mixin {
    return []ent.Mixin{
        mixin.AutoIncrementId{}, mixin.TimeAt{}, mixin.OperatorID{},
        mixin.IsEnabled{}, mixin.SortOrder{}, mixin.TenantID[uint32]{},
    }
}

func (CategoryDefaultFeature) Edges() []ent.Edge {
    return []ent.Edge{
        edge.From("category", Category.Type).
            Ref("default_features").
            Field("category_id").
            Required().Unique().
            Annotations(entsql.Annotation{OnDelete: entsql.Restrict}),
        edge.From("feature", Feature.Type).
            Ref("category_default_entries").
            Field("feature_id").
            Required().Unique().
            Annotations(entsql.Annotation{OnDelete: entsql.Restrict}),
    }
}

func (CategoryDefaultFeature) Indexes() []ent.Index {
    return []ent.Index{
        // 同一分类下同一特征不可重复
        index.Fields("tenant_id", "category_id", "feature_id").
            Unique().
            StorageKey("uix_tm_cat_default_features_tenant_cat_feat"),

        // 列表主查询
        index.Fields("tenant_id", "category_id", "sort_order").
            StorageKey("idx_tm_cat_default_features_tenant_cat_sort"),

        // 反向：某特征被哪些分类引用
        index.Fields("feature_id").
            StorageKey("idx_tm_cat_default_features_feature"),

        index.Fields("is_enabled").
            StorageKey("idx_tm_cat_default_features_enabled"),
        index.Fields("tenant_id").
            StorageKey("idx_tm_cat_default_features_tenant"),
    }
}
```

> Category schema 需追加反向 edge `default_features`；Feature schema 需追加反向 edge `category_default_entries`。两处都是已落地 schema 的小改动，详见 [04 §1.2](./04-后端实现设计.md)。

#### 约束（应用层兜底）

| # | 约束 | 实现层 |
|---|------|--------|
| C1 | `category_id` 节点必须 `level == 4` | Service 写入前校验 |
| C2 | `override_spec` 仅允许白名单字段 | Service 校验器（[04 §3](./04-后端实现设计.md)）|
| C3 | 写入时 `thing_features.reference_count++`；删除时 `reference_count--` | Service 事务 |
| C4 | `thingmodel_units.reference_count` 维护：仅当 override_spec.unit 或原 feature.spec.unit 有 unitId 且新增/删除一行时 ±1（同行内修改 unit 时旧 -1 新 +1） | Service 事务 |
| C5 | DB 层 `(tenant_id, category_id, feature_id)` 唯一 | DB 唯一索引 |

| 字段 | 类型 | 约束 | 说明 |

#### 约束

| # | 约束 | 实现层 |
|---|------|--------|
| C12 | `source=LOCAL` ↔ `ref_feature_id IS NULL`；`source∈{DEFAULT,GLOBAL}` ↔ `ref_feature_id IS NOT NULL` | Service 校验器 |
| C13 | `feature_snapshot` 必填；类型与 `feature_type` 一致（PropertySpec/EventSpec/ServiceSpec/RelationSpec） | Service 校验器 |

| C15 | 删除 source=GLOBAL 行 → `thing_features.reference_count--`；删除 source=DEFAULT 行不动（由 default_features 一侧维护） | Service 事务 |
| C16 | `(product_id, code)`、`(product_id, identifier)` 各自唯一 | DB 唯一索引 |
| C17 | LOCAL 特征 code 命名规则：建议前缀 `L-`（如 `L-PROP-0001`），与全局清单 `P-`/`E-`/`S-`/`R-` 区分；本期前端约定不强校验 | UI 提示 |
| C18 | 产品 status=PUBLISHED 后，禁止：增/删 feature 条目；修改 source 字段；修改 feature_snapshot 内非 override_spec 字段 | Service 校验器 |

## 3. `feature_snapshot` 与 `override_spec` 的 JSON 结构

### 3.1 `feature_snapshot`（完整快照）

直接复用 [特征管理 §3 FeatureSpec](../特征管理/09-特征数据模型设计.md)：

```jsonc
// 属性 (PropertySpec) 示例
{
  "dataType": "DOUBLE",
  "accessMode": "RW",
  "category": "measurement",
  "unit": {
    "unitId": 12,
    "unitCode": "celsius",
    "unitSymbol": "℃"
  },
  "constraints": { "min": -20, "max": 60, "step": 0.5 },
  "defaultValue": 25
}

// 事件 (EventSpec)、服务 (ServiceSpec)、关系 (RelationSpec) 同理
```

### 3.2 `override_spec`（稀疏覆写）

新增的 proto message `FeatureOverrideSpec`：

```protobuf
message FeatureOverrideSpec {
  // 白名单字段（均可选；空表示未覆写）
  ValueConstraints constraints  = 1;  // 覆写 min/max/step
  UnitRef          unit         = 2;  // 覆写单位
  google.protobuf.Value default_value = 3 [json_name="defaultValue"];
  string           display_name = 4 [json_name="displayName"];
  string           description  = 5;
  google.protobuf.BoolValue required = 6;
}
```

**effective_spec 计算（运行时）**：

```go
func EffectiveSpec(snapshot *FeatureSpec, override *FeatureOverrideSpec) *FeatureSpec {
    if override == nil { return snapshot }
    out := proto.Clone(snapshot).(*FeatureSpec)
    // 仅 property 才有 constraints/unit/defaultValue
    if p := out.GetProperty(); p != nil {
        if override.Constraints != nil { p.Constraints = override.Constraints }
        if override.Unit != nil        { p.Unit = override.Unit }
        if override.DefaultValue != nil{ p.DefaultValue = override.DefaultValue }
    }
    // displayName/description 在 SDK 层独立字段（非 spec 内），见 [03 §3]
    return out
}
```

### 3.3 为什么 displayName 不放 override_spec 里

> displayName 是**产品层"展示别名"**，与 spec（特征语义）正交。
> 实现上我们在 `thingmodel_category_default_features` / `thingmodel_product_features` 表中独立有 `display_name` 列；override_spec 内冗余存一份仅是为 proto 契约简洁。
> Service 写入时以**override_spec.displayName 优先**回填到独立列，列存查询用，JSON 存契约用。

## 4. `reference_count` 维护矩阵

| 操作 | thing_features.reference_count | thingmodel_units.reference_count |
|------|-------------------------------|----------------------------------|
| 新增 category_default_feature 行 | feature_id 对应行 +1 | snapshot.unit.unitId 对应行 +1 |
| 删除 category_default_feature 行 | feature_id 对应行 -1 | snapshot.unit.unitId 对应行 -1 |
| 修改 category_default_feature.override_spec.unit | 旧 unitId -1，新 unitId +1 | 同左 |
| 新增 product_feature source=GLOBAL | feature_id 对应行 +1 | snapshot.unit.unitId 对应行 +1 |
| 新增 product_feature source=DEFAULT | **不变**（默认模型一侧已 +1） | **不变** |
| 新增 product_feature source=LOCAL | **不变** | LOCAL 特征若 spec.unit.unitId 有值则 +1 |
| 删除 product_feature | 按 source 反向，与新增对称 | 同上 |
| 删除产品（CASCADE） | 遍历所有 product_features 按 source 处理 | 同上 |
| 修改 product_feature.override_spec.unit | 旧 unitId -1，新 unitId +1（不论 source） | — |

**事务一致性**：所有 ±1 操作必须与主表 CRUD 同事务（Repo 层 helper `IncFeatureRef / DecFeatureRef / SwapUnitRef`）。

## 6. 同 schema 改动清单

为支撑新表的反向 edge，**需要在以下已落地 schema 上追加 Edges**（不改字段，零迁移风险）：

```go
// internal/data/ent/schema/category.go
func (Category) Edges() []ent.Edge {
    return []ent.Edge{
        edge.To("children", Category.Type).From("parent").Field("parent_id").Unique().
            Annotations(entsql.Annotation{OnDelete: entsql.Restrict}),
        // ===== 新增 =====
        edge.To("default_features", CategoryDefaultFeature.Type).
            Annotations(entsql.Annotation{OnDelete: entsql.Restrict}),
        edge.To("products", Product.Type).
            Annotations(entsql.Annotation{OnDelete: entsql.Restrict}),
    }
}

// internal/data/ent/schema/feature.go
func (Feature) Edges() []ent.Edge {
    return []ent.Edge{
        // ===== 新增 =====
        edge.To("category_default_entries", CategoryDefaultFeature.Type).
            Annotations(entsql.Annotation{OnDelete: entsql.Restrict}),
    }
}
```

> `feature.product_features` 反向 edge 不建：product_feature.ref_feature_id 是弱引用，CASCADE/RESTRICT 在应用层 Service 维护，避免循环导入。
