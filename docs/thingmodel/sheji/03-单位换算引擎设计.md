# 03 · 单位换算引擎设计 ⭐

> 这是"模型属性单位管理"区别于普通字典 CRUD 的**核心技术点**。
> 《物模型属性单位清单》的"换算公式"列包含 5 种本质不同的换算形态，必须用统一的类型学承载，并在保证安全（无 `eval`）的前提下提供双向换算 API。

---

## 1. 换算类型学（从清单归纳）

通读清单 220+ 单位的换算公式，归纳为 **5 种换算类型**：

| 类型 | 枚举值 | 数学表达 | 清单占比 | 典型单位 |
|------|--------|----------|----------|----------|
| **线性** | `LINEAR` | `y = x · k` | ~85% | cm→m (`x*0.01`)、kPa→Pa、kW→W、km→m、min→s、kWh→J |
| **仿射** | `AFFINE` | `y = x · k + b` | <1% | ℃→K (`x+273.15`)、℉→K (`(x+459.67)*5/9`)、露点℃→K |
| **对数** | `LOGARITHMIC` | 非线性，不可线性换算 | ~2% | dB、dBm (`10^(x/10) mW`)、dB(A)、dB(C) |
| **条件** | `CONDITIONAL` | 依赖外部参数，无固定系数 | ~1% | ppm↔mg/m³（依赖气体分子量 M、温度 T） |
| **不可换算** | `NONE` | 无量纲/计数/无单位 | ~11% | pH、次/台/个/路、%(无量纲)、PF、η、"无单位" |

> **关键洞察**：`y = x · factor + offset` 这一个仿射模型即可覆盖 **所有可线性换算的单位**（LINEAR + AFFINE，合计约 86%）。剩余 14% 用类型标记排除出自动换算，仅做展示。**无需引入任意表达式求值。**

## 2. 公式模型

### 2.1 正向：单位值 → 基准值

```
base_value = source_value · factor + offset
```

- 基准单位自身：`factor = 1`, `offset = 0`（恒等）。
- 线性单位：`offset = 0`，退化为 `base = x · factor`。
- 仿射单位：`factor` 与 `offset` 均可能非 0/非 1。

### 2.2 反向：基准值 → 单位值

```
unit_value = (base_value - offset) / factor     （要求 factor ≠ 0）
```

### 2.3 任意两单位互转（同分类内）

设单位 A（factor=fA, offset=oA）、单位 B（factor=fB, offset=oB），均属可换算类型：

```
A → base :   b = a · fA + oA
base → B :   B = (b - oB) / fB

合并：  B = (a · fA + oA - oB) / fB
```

**示例**：50℃ → ℉
- celsius: fA=1, oA=273.15；fahrenheit: fB=5/9, oB=255.372
- ℉ = (50·1 + 273.15 − 255.372) / (5/9) = (67.778) / 0.5556 = **122.0 ℉** ✓

### 2.4 不可自动换算类型的处理

| 类型 | 引擎行为 |
|------|---------|
| `LOGARITHMIC` | 拒绝自动换算，返回错误码 `UNIT_LOGARITHMIC_NOT_CONVERTIBLE`；`formula_expr` 展示换算说明（如 dBm→mW: `10^(x/10)`） |
| `CONDITIONAL` | 拒绝自动换算，返回 `UNIT_CONDITIONAL_REQUIRES_PARAMS`；提示所需外部参数（如分子量 M、温度 T） |
| `NONE` | 拒绝，返回 `UNIT_NOT_CONVERTIBLE` |

## 3. Convert API 设计

### 3.1 单值换算

```
rpc Convert(ConvertUnitRequest) returns (ConvertUnitResponse)
```

**请求**：

```protobuf
message ConvertUnitRequest {
  // 源单位（二选一定位）
  oneof source_by { uint32 source_unit_id = 1; string source_unit_code = 2; }
  // 目标单位（二选一定位）
  oneof target_by { uint32 target_unit_id = 3; string target_unit_code = 4; }
  double value = 5;                       // 待换算的数值
  optional int32 precision = 6;           // 结果精度（小数位），缺省用目标单位 precision
}
```

**响应**：

```protobuf
message ConvertUnitResponse {
  double result = 1;                      // 换算结果
  string formula = 2;                     // 人类可读换算链，如 "50 ℃ → 323.15 K → 122.00 ℉"
  ConvertUnitStatus status = 3;           // OK / NOT_CONVERTIBLE / DIFFERENT_CATEGORY / ...
  string message = 4;                     // 非 OK 时的说明
  optional double base_value = 5;         // 经基准单位的中转值（调试/展示用）
}
```

### 3.2 校验链（Service 层）

```
Convert(value, src, dst):
  1. 解析 src/dst 为 Unit（按 id 或 code）
  2. 若任一不存在                  → 错误 UNIT_NOT_FOUND
  3. 若 category_id 不同           → 错误 UNIT_DIFFERENT_CATEGORY
  4. 若任一 conversion_type ∈ {NONE, LOGARITHMIC, CONDITIONAL}
                                   → 错误 UNIT_NOT_CONVERTIBLE（带 formula_expr 提示）
  5. 若 src.factor 或 dst.factor == 0 → 错误 UNIT_INVALID_FACTOR（数据异常，不应发生）
  6. 计算 base = value · src.factor + src.offset
  7. 计算 result = (base - dst.offset) / dst.factor
  8. 溢出/NaN/Inf 检查              → 错误 UNIT_OVERFLOW
  9. 按 precision 四舍五入
  10. 拼装 formula 字符串并返回
```

### 3.3 批量换算（前瞻，本期可选）

```protobuf
rpc ConvertBatch(ConvertBatchUnitRequest) returns (ConvertBatchUnitResponse)
// 用于设备历史数据归一化：一次提交 N 个 value，同一 src→dst
```

> 本期实现单值 `Convert`；`ConvertBatch` 作为前向兼容预留，可延后。

## 4. 写入校验规则（Create/Update 时强制）

在 Service 层 Create/Update 单位时强制以下一致性，避免脏数据破坏换算正确性：

| 规则 | 触发条件 | 错误 |
|------|----------|------|
| 基准单位系数固定 | `is_base == true` 但 `factor != 1` 或 `offset != 0` | `UNIT_BASE_FACTOR_INVALID` |
| 线性无偏移 | `conversion_type == LINEAR` 但 `offset != 0` | `UNIT_LINEAR_OFFSET_MUST_BE_ZERO` |
| 系数非零 | `factor == 0`（基准单位除外） | `UNIT_FACTOR_ZERO` |
| 同分类单位归属 | Create 时 `category_id` 必须存在且启用 | `UNIT_CATEGORY_NOT_FOUND` |
| 基准单位唯一 | 设置 `is_base = true` 时，该分类已有其它基准 | `UNIT_BASE_ALREADY_EXISTS`（事务内切换：先清旧再设新） |
| 删除约束 | `reference_count > 0` 时物理删除 | `UNIT_IN_USE_CANNOT_DELETE`（改为停用） |

> 这些规则集中在 `unit_service.go` 的私有校验方法，单元测试必须覆盖（见 [07 验收标准](./07-种子数据与实施计划.md)）。

## 5. 数值精度与溢出

| 关注点 | 处理 |
|--------|------|
| 运算类型 | 全程 `float64`（Go `float64`，proto `double`） |
| 舍入 | 结果按 `precision`（目标单位建议精度或请求覆盖值）`math.Round` 到指定小数位 |
| 中间值 | `base_value` 不预先舍入，仅最终结果舍入，避免误差累积 |
| 溢出 | 计算 `result` 后用 `math.IsNaN` / `math.IsInf` 检查，命中则返回 `UNIT_OVERFLOW` |
| 系数精度 | 种子数据的 `factor` 用清单原始值；循环小数（如 5/9=0.555...）存足够位数的 `double`（5/9 在 double 下精度足够工程用途） |

## 6. 为什么不用表达式引擎（拒绝 eval 的论证）

一个看似"更灵活"的方案是：存任意公式字符串 `"x*0.01"` / `"(x+459.67)*5/9"`，运行时用表达式引擎求值。**本设计明确拒绝此方案**，理由：

1. **安全**：任何动态求值（`eval`、`goja`、`expr`、Lua）都引入注入面。单位是系统基础数据，一旦被污染可影响所有设备数据归一化。`factor+offset` 是纯数值字段，无注入可能。
2. **正确性可证**：仿射变换的可逆性、链式复合性在数学上严格成立，易于写死、易于测试。任意表达式无法保证可逆、无法保证链式正确。
3. **可聚合**：系数存在 DB，未来可用 SQL 直接做批量归一化（`UPDATE ... SET v = v*factor+offset`），表达式则必须逐行走应用层。
4. **覆盖足够**：清单证明 86% 可换算单位可被仿射模型完整覆盖；剩余 14% 本就不可线性换算，存表达式也换算不了，只能展示——而展示用 `formula_expr` 纯文本即可。
5. **一致性**：与项目"结构化优于动态化"的整体风格一致（如权限用 Casbin 策略而非脚本、配置用 YAML 而非代码）。

> `formula_expr` 字段**仅用于前端展示**（给用户看"这个单位怎么换算"），**绝不参与计算**。

## 7. 换算链可视化（前端）

`ConvertUnitResponse.formula` 提供人类可读的换算链，前端在单位详情页/换算器中展示：

```
输入: 50 ℃
┌─────────┐     ×1 +273.15     ┌─────────┐     (−255.372)÷(5/9)     ┌─────────┐
│ 50 ℃    │ ─────────────────▶ │ 323.15 K│ ────────────────────────▶ │ 122 ℉   │
└─────────┘                    └─────────┘                            └─────────┘
```

前端（[06](./06-前端实现设计.md)）提供"单位换算器"小组件：选分类 → 选源单位 → 输入值 → 选目标单位 → 实时显示结果与换算链。
