# 07 · 扩展性设计 / kind 枚举扩展指南（v2.0）

> 本章回应 v2.0 第 3 条原则：
> "kind 字段本身就区分了系统/设备/空间等，将其可维护项枚举后续可扩展增加证书、文档、表单等等就可以了，不用引入新的字段了。"
>
> 本章替代 v1.0 的"`scope` + `Scope Validator`"双维度设计——v2.0 直接走 **kind 枚举扩展 + KindValidator 注册器** 的更简路径。

---

## 1. 核心思想

> **未来要支持新业务的分类（如文档、证书、表单、资讯、流程……），就在 `CategoryKind` 枚举里追加新值，并在 Service 层注册对应的 KindValidator 实现"该 kind 的特殊校验规则"——schema/表结构零改动。**

```
本期 kind = { SYSTEM, SPACE, FACILITY }     ← 都是物模型 4 层数字 code 强结构
未来 kind = { ..., DOCUMENT, CERTIFICATE, FORM, ARTICLE, ... }     ← 可能各自有不同规则
                  │
                  ▼
           不同的 KindValidator 实例
           （注册在 Service 层）
```

---

## 2. v2 vs v1 抽象方案对比

| 维度 | v1.0（scope + kind 双维度） | **v2.0（kind 单维度）** |
|------|-----------------------|------------------------|
| 数据库字段 | `scope` + `kind` | **仅 `kind`** |
| 唯一键 | `(tenant, scope, kind, code, level)` | **`(tenant, kind, code)`** |
| 概念个数 | 2 层概念（scope/kind），用户需理解嵌套 | 1 层概念，直接 |
| 元数据配置表 | 需要 `platform_category_scopes` 表 | **不需要** |
| 校验规则承载 | `ScopeValidator` 注册器 + 配置表驱动 | **`KindValidator` 注册器**（Go 代码直接注册） |
| 新增业务方接入工作量 | 1 行 SQL + 配置表写入 + 菜单 + 权限码 | **proto enum 追加 1 值 + Go 注册 1 个 Validator + 菜单 + 权限码** |
| 未实现业务的演进成本 | 高（先建空概念再使用） | **极低**（kind 枚举本来就要扩展） |

→ **核心洞察**：scope 抽象的"价值"是"同 scope 下有多 kind 互相联系"。本项目所有候选业务（物模型/文档/证书/表单）的分类都是**互不引用、各自独立的森林**——scope 没有第二个 kind 集合可承载，**抽象空转**。

**结论**：v2 用 kind 单维度承载所有业务，**简单且足够**。

---

## 3. 演进路径（三阶段）

### Stage 1：本期 — 物模型三 kind（当前设计 01–06 章）

- kind enum: `SYSTEM` / `SPACE` / `FACILITY`
- 唯一约束：所有 kind 都遵循"4 层 + 变长数字 code（2/4/6/8）"
- 单一 Service 内联校验逻辑

✅ 满足本期目标，零延期风险。

### Stage 2：业务扩展 — 引入第一个新 kind（如 DOCUMENT）

> 触发条件：业务方来要分类能力。

**改动量**（约 0.5–1 人日，比 v1.0 的 Stage 2 节省 50%）：

1. **proto enum 追加**：
   ```protobuf
   enum CategoryKind {
     ...existing...
     DOCUMENT = 10;
   }
   ```
2. **ent enum 同步**：
   ```go
   field.Enum("kind").NamedValues(
       ...existing...,
       "Document", "DOCUMENT",
   )
   ```
   重新 `gow ent` + `gow api`。
3. **Service 注册 DocumentKindValidator**（如该业务规则与物模型不同，例如允许 1-6 层、code 是 slug 而非数字）：
   ```go
   type DocumentKindValidator struct{}
   func (DocumentKindValidator) Kind() thingmodelV1.CategoryKind { return thingmodelV1.CategoryKind_DOCUMENT }
   func (DocumentKindValidator) Validate(ctx context.Context, c *thingmodelV1.Category, parent *ent.Category) error {
       // 文档分类规则：
       // - code 允许 [a-z0-9_-]{1,64}
       // - level 1..6
       // - 父子关系：parent_id 即父级，无 code 前缀要求
       if !slugRegex.MatchString(c.GetCode()) { return ... }
       if c.GetLevel() < 1 || c.GetLevel() > 6 { return ... }
       // ... 等等
       return nil
   }

   // Service 启动时注册
   service.RegisterKindValidator(&DocumentKindValidator{})
   ```
4. **菜单 + 权限码**：在 `default_data.go` 追加菜单，自动派生 `category:document:view` 等权限码。

**数据兼容**：现有物模型分类全部保持 `kind ∈ {SYSTEM, SPACE, FACILITY}`，不受影响。新 kind=DOCUMENT 与之共表共索引但互不干扰。

### Stage 3：拆表 — 仅在某 kind 数据量爆炸（>1000 万节点）时考虑

> 触发条件：单 kind 数据量超过千万级（极端场景，预计 3–5 年内不需要）。

- 把超大 kind 拆出独立表（如 `document_categories`），共享 Service/前端组件。

## 4. KindValidator 注册器设计

### 4.1 接口定义

```go
// KindValidator 每个 kind 一个，定义该 kind 的校验规则
type KindValidator interface {
    Kind() thingmodelV1.CategoryKind
    
    // Validate 校验创建/更新参数
    // parent 可为 nil（level=1 时）；返回 nil 表示通过
    Validate(ctx context.Context, c *thingmodelV1.Category, parent *ent.Category) error
}

// Registry 全局注册中心（Wire 注入）
type ValidatorRegistry struct {
    validators map[thingmodelV1.CategoryKind]KindValidator
}

func NewValidatorRegistry() *ValidatorRegistry {
    r := &ValidatorRegistry{validators: map[thingmodelV1.CategoryKind]KindValidator{}}
    // 本期注册：
    r.Register(&ThingmodelKindValidator{})  // 同时承担 SYSTEM/SPACE/FACILITY 三 kind（规则相同）
    return r
}

func (r *ValidatorRegistry) Register(v KindValidator) { r.validators[v.Kind()] = v }

func (r *ValidatorRegistry) Get(k thingmodelV1.CategoryKind) (KindValidator, bool) {
    v, ok := r.validators[k]
    return v, ok
}
```

### 4.2 本期实现：ThingmodelKindValidator

封装 [04 §6](./04-后端实现设计.md#6-关键校验汇总单元测试一对一映射) 的 12 项校验：

```go
type ThingmodelKindValidator struct{}

// 本期 Validator 同时服务 SYSTEM/SPACE/FACILITY 三 kind，规则相同
// 注册时按 kind 重复注册即可
func (v *ThingmodelKindValidator) Kind() thingmodelV1.CategoryKind {
    return thingmodelV1.CategoryKind_SYSTEM  // 占位，实际 Register 时 3 次注册
}

func (v *ThingmodelKindValidator) Validate(ctx context.Context, c *thingmodelV1.Category, parent *ent.Category) error {
    code := c.GetCode()
    if !digitsOnly.MatchString(code) {
        return thingmodelV1.ErrorCategoryCodeFormatInvalid("code must be all digits")
    }
    lvl := c.GetLevel()
    if lvl < 1 || lvl > 4 {
        return thingmodelV1.ErrorCategoryLevelInvalid("level must be 1..4")
    }
    if uint32(len(code)) != lvl*2 {
        return thingmodelV1.ErrorCategoryCodeLengthMismatch(...)
    }
    if parent != nil {
        if !strings.HasPrefix(code, parent.Code) || len(code) != len(parent.Code)+2 {
            return thingmodelV1.ErrorCategoryCodePrefixMismatch(...)
        }
    }
    return nil
}
```

> 注册时调用 `r.Register(v)` 三次（kind 不同但 v 是同实例）——也可以稍微重构成 `multiKindValidator`，本期为了简单选择重复注册。

### 4.3 Service 改造（最小化）

`CategoryService.validateCreate` 改为：

```go
func (s *CategoryService) validateCreate(ctx context.Context, c *thingmodelV1.Category) error {
    // 通用校验（所有 kind 都必须满足）
    if c.GetKind() == thingmodelV1.CategoryKind_CATEGORY_KIND_UNSPECIFIED {
        return thingmodelV1.ErrorBadRequest("kind required")
    }
    if c.GetName() == "" {
        return thingmodelV1.ErrorBadRequest("name required")
    }
    // parent 一致性校验（kind/level/tenant）
    var parent *ent.Category
    if c.GetLevel() > 1 {
        // ... 查父节点、校验 kind/level 一致 ...
    }
    // 委托给 kind 专属 Validator
    validator, ok := s.registry.Get(c.GetKind())
    if !ok {
        return thingmodelV1.ErrorBadRequest("unsupported kind")
    }
    return validator.Validate(ctx, c, parent)
}
```

> 本期 ValidatorRegistry 即使不引入，所有校验内联在 Service 里也完全可行。**Stage 2 演进时再抽出 Registry 即可**——本期不必预先抽象。

## 5. 决策表：业务方接入新分类应走哪条路？

| 业务方诉求 | 应选路径 | 工作量 |
|-----------|----------|--------|
| 想要"分类树 + 增删改查 + 启停 + 排序" | **复用本模块**（追加 kind 枚举值 + 注册 Validator） | 0.5–1 人日 |
| 分类规则与物模型相同（4 层 + 数字 code） | 追加 kind + 复用 `ThingmodelKindValidator` | < 0.5 人日 |
| 分类规则不同（如 slug code、最多 6 层） | 追加 kind + 写专属 KindValidator | 1 人日 |
| 分类需要**额外业务字段**（如证书需"颁发机构""有效期模板"） | 复用本模块 + **业务方自建扩展表**（FK→category_id） | 复用 + 加 1 张关联表 |
| 分类需要**完全不同的字段集**或**特殊查询模式** | **拆表自建**，参考本设计范式 | 完整 6 章重做 |

> 多数情况会落到**前两条**——这就是 v2 的价值：**让 N 个业务方共享一套基础设施**，且**接入成本远低于 v1**。

---

## 6. 本期需要预留的钩子（v2 视角：几乎为 0）

v1 在本期建议预留 `scope` 字段。**v2 不需要预留任何东西**——因为：

- ✅ `kind` 枚举本来就支持追加值（proto enum 天然兼容）
- ✅ 唯一键 `(tenant, kind, code)` 已经能容纳任意 kind 共表
- ✅ Service 当前内联校验逻辑，Stage 2 演进时再抽出 Registry，**0 数据迁移、0 索引重建**

唯一建议（可选）：

| 钩子 | 操作 | 收益 |
|------|------|------|
| **Service 校验逻辑函数化** | 把 `validateCreate` 里的具体规则抽成 `validateThingmodelKind(...)` 一个私有方法 | Stage 2 抽出 KindValidator 接口时只需把私有方法实现到 struct 即可，零业务影响 |

**不**预留的事情：

- ❌ 不预先建 `KindValidator` 接口和 Registry —— 本期只有"物模型一种规则"，注册器是过度设计。
- ❌ 不预先在 kind 枚举里加 `DOCUMENT/CERTIFICATE/FORM` 占位 —— 未启用的枚举值反而误导阅读者。
- ❌ 不预先建配置表 —— 完全用 Go 代码即可，比 v1 的 `platform_category_scopes` 表更简单。

## 7. 案例草图：文档分类如何接入（Stage 2 之后）

仅作示意，无需本期实现：

```protobuf
// proto 追加（5 行改动）
enum CategoryKind {
  ...existing...
  DOCUMENT = 10;
}
```

```go
// ent schema 同步追加（1 行改动）

    ...existing...,
    "Document", "DOCUMENT",

```

```go
// 注册 Validator（新增文件 50 行）
type DocumentKindValidator struct{}
func (v *DocumentKindValidator) Kind() thingmodelV1.CategoryKind {
    return thingmodelV1.CategoryKind_DOCUMENT
}
func (v *DocumentKindValidator) Validate(ctx context.Context, c *thingmodelV1.Category, parent *ent.Category) error {
    // 文档分类规则：自由 slug、1-6 层
    if !slugRegex.MatchString(c.GetCode()) {
        return errors.New("code must be slug [a-z0-9_-]{1,64}")
    }
    if c.GetLevel() < 1 || c.GetLevel() > 6 {
        return errors.New("level must be 1..6")
    }
    // 父子关系：仅校验 parent_id 存在，不要求 code 前缀
    return nil
}

// Wire 注册
service.RegisterKindValidator(&DocumentKindValidator{})
```

## 8. 结论

- v2 用 `kind` 单维度承载多业务分类，**比 v1 的 scope+kind 双维度更简单**。
- 本期**不需要预留任何字段或表**，因为 kind 枚举天然支持追加。
- 当**第一个非物模型业务方**来要分类能力时：proto 加 1 行 enum、ent 加 1 行、Go 写 1 个 KindValidator（如规则与物模型相同则连这步都省），完成。
- 这是经典的"YAGNI + 接口足够通用"组合：当前最小代价、未来低成本扩展。

### 9. 与 v1.0 第 7 章的差异总结

| 主题 | v1.0 第 7 章 | **v2.0 第 7 章** |
|------|-------------|-------------------|
| 扩展机制 | 引入 `scope` 字段 + `platform_category_scopes` 配置表 + `ScopeValidator` 注册器 | **直接扩展 `kind` 枚举 + `KindValidator` 注册器** |
| 本期需预留的字段 | 1 个（`scope`） | **0 个** |
| Stage 2 改动量 | 1–1.5 人日（含 schema/index 迁移） | **0.5–1 人日**（仅 proto enum + 注册器抽出，无数据迁移） |
| 新业务方接入成本 | 1 行 SQL 配置 + 菜单 + 权限码 | **proto 1 改 + Go 1 个 Validator + 菜单 + 权限码** |
| 概念复杂度 | scope（业务域）+ kind（业务域内子类） | **kind**（统一） |
| 是否需要配置驱动 | 是（platform_category_scopes 表） | **否**（Go 代码直接注册） |

→ v2 是 v1 的**纯简化**，没有任何能力损失。
