## 1. 背景与问题

### 1.1 现状回顾

当前 `thingmodel` 领域采用三层架构：

| 层 | 表 | 承载内容 |
|----|----|----------|
| L0 特征字典 | `thing_features` | 全局可复用的"特征定义"——含**完整 `FeatureSpec`**（dataType / accessMode / constraints / enumItems / boolLabels / textMaxLength / structFields / arraySpec / unit 等） |
| L1 分类默认模型 | `thingmodel_category_default_features` | 分类（level=4）选取一批 `thing_features`，可挂 **`FeatureOverrideSpec`** 稀疏覆写（白名单 6 字段） |
| L2 产品模型 | `thingmodel_product_features` | 创建产品时按勾选从 L0/L1 拷贝完整 `feature_snapshot`，可再挂 `FeatureOverrideSpec` 覆写 |

其中 `FeatureOverrideSpec`（白名单）只允许覆写：

```text
constraints  · unit  · defaultValue  · displayName  · description  · required
```

### 1.2 现状的不合理之处

用户在使用"特征管理"功能（`/thingmodel/feature`）配置具体特征时观察到：

> **不同产品即使特征属性名称相同，结构化约束可能完全不同。**

| 反例 | 同名/同 identifier 在不同产品下的差异 |
|------|--------------------------------------|
| "压力值" `pressure` | A 产品：`double / 0~1.6 MPa`；B 产品：`double / 0~25 MPa`；C 产品：`int / 0~999 kPa` |
| "运行模式" `runMode` | 风机盘管：`{0:停机,1:制冷,2:制热,3:送风}`；冷水机组：`{0:停机,1:手动,2:自动,3:远程}` |
| "故障代码" `faultCode` | 不同厂商：`int` vs `text`；不同位宽 |
| "进出水温度" `inletOutletTemp` | 冷机：`struct{inlet,outlet}`；末端：`double` 单值 |

即便我们已经提供了 `FeatureOverrideSpec`，**白名单只覆盖了 6 个字段**（连最常见的"数据类型本身不同""枚举项内容不同""struct 子字段不同"都覆写不了）。结果就是：

1. **特征 spec 形同虚设**：L0 spec 里的 enumItems / structFields / arraySpec / textMaxLength / boolLabels 在 L1/L2 一旦需要差异化，根本无法通过 override 实现，只能"再造一个特征"，与"全局特征字典"的初衷冲突。
2. **L0 字段语义错位**：`thing_features.data_type` 应该是"该特征推荐/常见的数据类型"，但目前是"该特征的真实数据类型"——耦合了产品语义。
3. **校验逻辑割裂**：L0 已经按 `data_type` 校验 enumItems/structFields 必填，但 L1/L2 用快照拷贝走后，又靠一套"白名单 override"做局部修订，两套校验难以保证最终 effective spec 的完整性。
4. **种子数据膨胀**：清单中"通用属性"被定义了一遍"参考 spec"，但任何产品要用它都要重写约束，相当于种子 spec 只是文档样例，不是可直接消费的数据。

### 1.3 一句话定性

> **"结构化约束"是"产品对该特征的具体实现细节"，不是"特征本身的属性"。**
> 当前模型把"实现细节"放到了"字典"层，再用一套残缺的 override 试图修订——方向倒置。

---

## 2. 设计目标

1. **L0 瘦身**：`thing_features` 只承载特征的**身份/语义骨架**，不承载任何"具体取值如何被约束"的信息。
2. **L1 自洽**：`thingmodel_category_default_features` 作为"分类推荐模型"，**完整定义**该分类典型产品的特征约束（不再依赖 L0 的约束、不再是稀疏覆写）。
3. **L2 自洽**：`thingmodel_product_features` 作为产品的**最终物模型**，完整定义产品的特征约束（可从 L1 拉取默认值后自由编辑，包括 dataType / enumItems / structFields）。
4. **校验单点**：spec 完整性校验（V1–V17）从 L0 上移到 L1/L2；L0 仅做骨架校验。
5. **契约最小破坏**：proto `FeatureSpec` / `ParamSpec` / 各子结构体形状不变，只是承载位置从 `Feature.spec` 移到 `CategoryDefaultFeature.spec` / `ProductFeature.spec`。
6. **数据迁移**：现有 L0 spec 内容**整体迁移**为 L1 的种子数据（按"通用 → level=4 分类"映射）。

---

## 3. 候选方案对比

### 方案 A：彻底下沉（推荐 ✅）

L0 不再有 `spec` 列、不再有 `data_type / access_mode / event_level / call_mode / relation_type` 等"特化抽取列"中**与具体取值结构相关**的字段。

| `thing_features` 保留 | `thing_features` 移除 |
|-----------------------|-----------------------|
| `feature_type` (PROPERTY/EVENT/SERVICE/RELATION) | `data_type` |
| `code` / `identifier` / `name` / `name_en` / `description` | `access_mode` |
| `applicable_scope` | `event_level` |
| **`recommended_unit_category_id`** (新增，可选)：单位**所在物理量分类**（如"压力"），不指定具体单位 | `call_mode` |
| **`semantic_tag`** (新增，可选)：语义标签（如 `pressure` / `temperature` / `runMode`），仅用于前端检索分组 | `relation_type` |
| `is_enabled` / `sort_order` / `tenant_id` / 审计字段 | `spec` (整列删除) |

L1/L2 完整承载：
- `CategoryDefaultFeature.spec` 字段类型从 `FeatureOverrideSpec` 升级为 **完整 `FeatureSpec`**（与原 L0 `spec` 同形）。
- `ProductFeature.feature_snapshot` 字段保持 `FeatureSpec`，但**初始值来源变为 L1**（不是 L0），且**全字段可编辑**（不再有 override 限制）。
- `ProductFeature.override_spec` 字段**整体删除**（不再需要——产品直接编辑 snapshot 本身）。

**优点**：
- 概念清晰：特征=身份，模型=约束。
- 校验集中：所有"取值结构"校验只发生在 L1/L2。
- 前端 UI 简化：特征管理页只剩骨架表单，不再有按 dataType 切换的动态约束面板。

**缺点**：
- 一次性破坏性较大：L0 数据/proto/前端表单/种子全部要改。
- 失去"全局默认 spec"的便利：建一个新分类下的新特征时，模型层必须从零填 spec（缓解：见 §6.2"模板/复制"能力）。

### 方案 B：扩大白名单（次选）

保留 L0 spec 现状，但把 `FeatureOverrideSpec` 升级为**全字段允许覆盖**（实际等价于"完整 FeatureSpec 替换"）：

```protobuf
message FeatureOverrideSpec {
  // 全字段允许覆写，但语义上只在非空时生效
  optional DataType data_type = 1;
  optional AccessMode access_mode = 2;
  optional ValueConstraints constraints = 3;
  optional UnitRef unit = 4;
  repeated EnumItem enum_items = 5;     // 注意：repeated 字段如何表达"未覆写"
  optional BoolLabels bool_labels = 6;
  ...
}
```

**优点**：
- L0 数据无需迁移，proto 兼容，前端旧表单可继续工作。
- 渐进式：可分模块切换到"自定义"。

**缺点**：
- `repeated` 字段无法用 proto 区分"未设置"和"设置为空数组"——只能引入 `presence_mask` 或 `oneof` 包装，复杂度高。
- 把"覆写"与"重定义"混在同一 message，语义模糊（"我是想清空 enumItems 还是没覆盖？"），前端难以表达。
- 双重存储（L0 全字段 + L1 全字段）数据冗余、易漂移。

### 方案 C：保留 L0 spec 作为"参考模板"（折中）

L0 `spec` 列保留但语义改为 **"参考模板（read-only template）"**：
- 不参与任何运行时约束计算。
- 仅在创建 L1/L2 条目时作为"初始填充"。

实现上：
- L1 `CategoryDefaultFeature.spec` 升级为完整 `FeatureSpec`（与方案 A 一致）。
- 创建 L1 条目时，后端从 L0 `spec` 拷贝为初值；之后两者独立。
- L0 spec 可继续在特征管理页编辑，但仅影响"未来新建"的 L1/L2 条目（不回写历史）。

**优点**：
- 平滑过渡：种子数据无需重写，L0 仍可作为"清单模板库"。
- 用户教育成本低：特征管理页保留约束表单，只是改提示文案为"模板默认值"。

**缺点**：
- 概念仍有歧义：用户在 L0 修改后会困惑"为什么我改了已有产品没变？"
- 仍存在数据冗余（L0 模板 + L1 实例双份）。

### 方案对比结论

| 维度 | 方案 A（下沉） | 方案 B（扩大白名单） | 方案 C（L0 模板化） |
|------|:---:|:---:|:---:|
| 概念清晰度 | ⭐⭐⭐ | ⭐ | ⭐⭐ |
| 实施复杂度 | 中 | 高（proto/校验都复杂） | 低 |
| 数据冗余 | 无 | 严重 | 中等 |
| 用户认知成本 | 中（一次性转变） | 高（持续混淆） | 低 |
| 长期可维护性 | ⭐⭐⭐ | ⭐ | ⭐⭐ |
| 推荐 | ✅ | ✗ | 备选 |

> **推荐采用方案 A**；若评审认为破坏性过大、希望分两阶段，可先按方案 C 上线一个迭代（数据/前端兼容），下一迭代再清理 L0 spec 列，最终达到方案 A 形态。

### 4.1 数据模型变更

#### 4.1.2 `thingmodel_category_default_features`（增强）

```diff
 type CategoryDefaultFeature struct{ ent.Schema }

 func (CategoryDefaultFeature) Fields() []ent.Field {
     return []ent.Field{
         field.Uint32("category_id"). ... ,
```

变化要点：
- `override_spec` 列**改名/换类型**：`override_spec (FeatureOverrideSpec)` → `spec (FeatureSpec)`，承载完整约束。
- 新增冗余抽取列（与 L0 现有同名列等价），用于列表筛选与索引。

#### 4.1.3 `thingmodel_product_features`（简化）

```diff
 type ProductFeature struct{ ent.Schema }

 func (ProductFeature) Fields() []ent.Field {
     return []ent.Field{
         field.Uint32("product_id"). ... ,
         field.Enum("source"). ... ,            // DEFAULT / GLOBAL / LOCAL（语义不变）
         field.Uint32("ref_feature_id"). ... ,

         field.Enum("feature_type"). ... ,
         field.String("code"). ... ,
         field.String("identifier"). ... ,
         field.String("name"). ... ,
         ...

-        // 完整快照
-        field.JSON("feature_snapshot", &thingmodelV1.FeatureSpec{}). ... ,
-        // 稀疏覆写
-        field.JSON("override_spec", &thingmodelV1.FeatureOverrideSpec{}). ... ,

+        // 完整 spec（产品最终物模型，全字段可编辑）
+        field.JSON("spec", &thingmodelV1.FeatureSpec{}).
+            Comment("产品完整 FeatureSpec / Full feature spec at product scope"),

         field.Enum("data_type"). ... ,
         field.Enum("access_mode"). ... ,
         field.Enum("event_level"). ... ,
         field.Enum("call_mode"). ... ,
         field.String("relation_type"). ... ,
     }
 }
```

变化要点：
- `feature_snapshot` + `override_spec` 双字段**合并为单一 `spec` 字段**。
- "有效 spec = snapshot ⊕ override" 的合并计算逻辑（`EffectiveSpec`）随之删除。
- `source` 含义不变，但语义改为"约束初值的来源"：
  - `DEFAULT`：从 L1 `category_default_features.spec` 拷贝初值；
  - `GLOBAL`：勾选了某个 `thing_feature` 骨架，spec 需在产品侧从零或从模板填写；
  - `LOCAL`：完全自定义。
- `ref_feature_id` 含义变为"骨架来源"，与运行时约束无关。

### 4.2 Proto 变更

#### 4.2.1 `feature.proto`

- 保留：所有 spec 结构（`FeatureSpec` / `PropertySpec` / `EventSpec` / `ServiceSpec` / `RelationSpec` / `ParamSpec` / `UnitRef` / `ValueConstraints` / `EnumItem` / `BoolLabels` / `ArraySpec` / `EntityRef`）。
- `message Feature`：
  - **删除**：`data_type` / `access_mode` / `event_level` / `call_mode` / `relation_type` / `spec` 字段。
  - **新增**：`recommended_unit_category_id` / `semantic_tag` 字段。
- `ValidateSpec` RPC：保留，但移动到 `CategoryDefaultFeatureService` 与 `ProductFeatureService`（即谁承载 spec 谁提供校验）。
- `ImportFeatureRow`：删除 `spec_json` 字段；保留 `feature_type` / `code` / `identifier` / `name` 等骨架字段。

#### 4.2.2 `feature_override_spec.proto`

- **整文件删除**。`FeatureOverrideSpec` message 弃用。

#### 4.2.3 `category_default_feature.proto`（新增/调整）

```protobuf
message CategoryDefaultFeature {
  optional uint32 id = 1;
  optional uint32 category_id = 2 [json_name = "categoryId"];
  optional uint32 feature_id = 3 [json_name = "featureId"];

  optional FeatureSpec spec = 4 [json_name = "spec"];   // 完整 spec
  optional string display_name = 5 [json_name = "displayName"];

  optional DataType   data_type   = 10 [json_name = "dataType"];
  optional AccessMode access_mode = 11 [json_name = "accessMode"];
  optional EventLevel event_level = 12 [json_name = "eventLevel"];
  optional CallMode   call_mode   = 13 [json_name = "callMode"];
  optional string     relation_type = 14 [json_name = "relationType"];

  optional bool is_enabled = 30;
  optional uint32 sort_order = 31;
  optional uint32 tenant_id = 100;
  // ... 审计字段
}
```

#### 4.2.4 `product_feature.proto`（调整）

- 字段层面把 `feature_snapshot` + `override_spec` 合为 `spec`（`FeatureSpec`）。
- 不再暴露 `EffectiveSpec` 计算接口。

### 4.3 Service / 业务逻辑变更

#### 4.3.1 创建/编辑特征（L0）

- 表单仅含：`feature_type` / `code` / `identifier` / `name` / `name_en` / `description` / `applicable_scope` / `recommended_unit_category_id` / `semantic_tag` / `is_enabled` / `sort_order`。
- 校验：骨架完整性（code 唯一、identifier 唯一、feature_type 必填）。
- **不再有 spec 校验**（V1–V17 移走）。

#### 4.3.2 创建/编辑分类默认模型（L1）

- 表单：选定 `feature_id` 后展开**完整 spec 编辑器**（按 `feature.feature_type` 渲染 PropertySpec / EventSpec / ServiceSpec / RelationSpec 表单）。
- 校验：V1–V17 全部在此处执行（前后端各一份）。
- 单位维护：`spec.unit.unitId` 变更时维护 `thingmodel_units.reference_count`。

#### 4.3.3 创建产品 / 编辑产品特征（L2）

- 勾选分类默认模型：拷贝 L1 `spec` 作为产品 `spec` 初值（`source=DEFAULT`）。
- 勾选全局特征骨架：拷贝 L0 骨架（`source=GLOBAL`），`spec` 需用户从零填写或从其它产品/分类模板复用。
- 新建本地特征：`source=LOCAL`，`spec` 完全由用户输入。
- **全字段可编辑**：dataType / enumItems / structFields 等都可改。
- 校验：V1–V17 全部在此处执行。
- `EffectiveSpec(snapshot, override)` 函数及调用链整体移除。

#### 4.3.4 reference_count 维护矩阵更新

| 操作 | thing_features.reference_count | thingmodel_units.reference_count |
|------|:---:|:---:|
| 新增 L1 行（任意特征） | feature_id +1 | spec.unit.unitId +1（含递归 params） |
| 删除 L1 行 | feature_id -1 | spec.unit.unitId -1 |
| 修改 L1.spec（单位变化） | 不变 | 旧 unitId -1 / 新 unitId +1 |
| 新增 L2.source=DEFAULT 行 | 不变（已由 L1 计入） | 不变（同上） |
| 新增 L2.source=GLOBAL 行 | feature_id +1 | spec.unit.unitId +1 |
| 新增 L2.source=LOCAL 行 | 不变 | spec.unit.unitId +1（若有） |
| 删除/修改 L2 行 | 对称反向 | 对称反向 |

> 关键变化：原矩阵中 `snapshot.unit ↔ override.unit` 双源合并的复杂度消失，因为 spec 已经是单一来源。

### 4.4 前端变更

#### 4.4.1 特征管理页（`/thingmodel/feature`）

- 删除：按 `data_type` / `access_mode` / `event_level` / `call_mode` 的筛选下拉。
- 删除：右侧动态 Spec 表单（PropertySpec/EventSpec/ServiceSpec/RelationSpec 表单组件）。
- 新增：`recommended_unit_category_id`（单位分类选择器，从单位管理读取）。
- 新增：`semantic_tag` 自由文本/标签输入。

#### 4.4.2 分类默认模型抽屉/页

- 新增：完整 Spec 编辑器（迁移自原特征管理的动态表单）。
- 保留：`display_name` / `sort_order` / `is_enabled`。

#### 4.4.3 产品详情页

- "Feature 抽屉"（`ProductFeatureDrawer`）：合并 `feature_snapshot` 与 `override_spec` 两个编辑区为单一 `spec` 编辑器；移除"覆写白名单"提示。
- 新增按钮："从分类默认模型拉取最新约束"（一次性覆盖 spec）、"从其它产品复制 spec"。

### 4.5 种子数据迁移

#### 4.5.1 现状

- 现有 `thingmodel_features` 行约 300+（来自《物模型特征清单》），每行含完整 spec。
- 现有 `thingmodel_category_default_features` 行（若已落地）含稀疏 `override_spec`。
- 现有 `thingmodel_product_features` 行含 `feature_snapshot` + `override_spec`。

#### 4.5.2 迁移策略

> 见 §5 数据迁移脚本。核心思想：

1. 把每行 L0 `spec` 拷贝到对应的"通用兜底分类（level=4）"的 L1 行——为每个 `applicable_scope` 取值生成一条 L1。
2. L1 已有 `override_spec` 行：与对应 L0 `spec` 合并（应用现有 `EffectiveSpec` 算法），结果写入新 `spec` 列；删除 `override_spec` 列。
3. L2 已有 `feature_snapshot` + `override_spec` 行：合并为 `spec` 列。
4. 最后 `ALTER TABLE thingmodel_features DROP COLUMN spec, data_type, access_mode, event_level, call_mode, relation_type;`

## 5. 实施步骤（建议两阶段）

### 阶段 1：模型层"承载完整 spec"上线（向后兼容）

> 目标：L1/L2 可独立承载完整 spec，但不删除 L0 spec 列；保证现有 L1/L2 流程不变。

1. **Proto**
   - 新增 `CategoryDefaultFeature.spec`（`FeatureSpec`）字段（与原 `override_spec` 共存）。
   - `ProductFeature` 新增 `spec` 字段（与 `feature_snapshot` + `override_spec` 共存）。
2. **Ent Schema**：相应新增列（NULLABLE）；运行 `make ent`。
3. **Service**：
   - L1/L2 写入逻辑双写（新 `spec` + 旧 `override/snapshot`）。
   - 读取优先使用新 `spec`，缺失时 fallback 到旧组合。
4. **前端**：
   - L1/L2 抽屉切换到"完整 Spec 编辑器"模式（隐藏旧 override UI）。
   - L0 特征管理页保留现状，但表单顶部加横幅提示"约束配置即将下沉到模型层"。
5. **数据迁移脚本（v1）**：把现有 L1/L2 的 `override_spec`/`snapshot` 合并写入新 `spec` 列。

### 阶段 2：L0 spec 列摘除（破坏性）

> 前置条件：所有租户的 L1/L2 已完成迁移，且观察期 ≥ 1 个迭代无回滚需求。

1. **Proto**：
   - 删除 `Feature.spec` / `Feature.data_type` 等 5 个特化列字段（按 proto reserved 规则保留 tag 号防回收）。
   - 删除 `feature_override_spec.proto`。
   - 删除 `CategoryDefaultFeature.override_spec` / `ProductFeature.feature_snapshot` + `ProductFeature.override_spec`。
2. **Ent Schema**：删除对应列；运行 `make ent` + DB 迁移 `ALTER TABLE`。
3. **Service**：删除 `EffectiveSpec`、双写、fallback 逻辑；spec 校验器集中到 `category_default_feature_service` / `product_feature_service`。
4. **前端**：移除 L0 特征管理页的 Spec 编辑器，仅保留骨架表单。
5. **种子数据**：重新生成 `14-特征种子数据-insert.sql` 与 `14-特征种子导入模板.xlsx`，去掉 spec 列；新增 `15-分类默认模型种子.sql`（承载原 L0 spec 内容）。

### 阶段 2 风险阀

- DB 迁移用"先 `RENAME COLUMN` 一个迭代、再 `DROP`"两步走，期间保留回滚窗口。
- 前端发布时灰度（按租户 feature flag）。

## 6. 设计抉择记录（ADR-lite）

### 6.1 为什么 L0 不留"推荐 spec"模板

考虑过让 L0 保留一个"参考默认 spec"（方案 C），但讨论后否决：

- 一旦 L0 spec 仍存在，用户难以避免在 L0 编辑——"为什么我改了不生效？"心智负担更重。
- 真正需要"模板复用"的场景应通过 §6.2 的模板库能力解决，而不是在字典层强塞一份默认值。

### 6.2 模板/复制能力（弥补"L0 没有 spec"的便利损失）

新增辅助能力（不属于本 CR 强依赖，可后续实现）：

1. **"复制自其它分类/产品"**：L1/L2 抽屉中提供"从已有的同 `feature_id` 条目复制 spec"按钮。
2. **特征 spec 库（独立小功能）**：在特征管理页提供"附加模板"子页（与 L0 主表分离）——用户可为某 feature_id 维护 1..N 个"模板 spec"（命名），L1/L2 创建时可挑选。表为 `thing_feature_spec_templates(feature_id, name, spec)`。

> 模板库是"用户自助"的，避免污染字典层。

### 6.3 是否保留 `applicable_scope`

保留。它描述的是"该特征适用于哪些设备类别"——这是骨架语义（"压力值"适用于"冷机/锅炉/泵"），与具体约束无关。

### 6.4 `semantic_tag` 的取值约束

`semantic_tag` 作为软性检索字段，不约束取值，但建议在前端提供下拉建议（取自现有所有 tag 的 distinct）。

### 6.5 `recommended_unit_category_id` 与 `spec.unit` 的关系

- `recommended_unit_category_id` 是"物理量分类"，指向单位管理的`thingmodel_unit_categories`（如"压力"），不指定具体单位。
- L1/L2 编辑 spec 时，单位选择器**可按此分类预过滤**（仅展示该分类下的单位），但用户也可切换分类。
- 这弱约束保留了"压力值"特征只配压力单位的指导意义，但不强制。
