## 1. 领域模型

```
┌──────────────────────────────────────────────────────────────────────┐
│                       thing_feature（统一特征表）                      │
│                                                                      │
│  公共字段（所有类型共享，9 项）                                         │
│  • feature_type  (PROPERTY / EVENT / SERVICE / RELATION)             │
│  • code          (P-RUN-0001 / E-GEN-0001 / S-GEN-0001 ...)          │
│  • identifier    (powerSwitch / deviceOnline / turnOn ...)           │
│  • name / name_en                                                    │
│  • description                                                       │
│  • applicable_scope  (适用设备：暖通/给排水/... 或 "通用")             │
│  • is_enabled / sort_order / tenant_id + 审计                        │
│                                                                      │
│  差异字段（按 feature_type 解读，JSON）                                 │
│  • spec  (JSONB —— 四类各自的结构，见 §3)                              │
│                                                                      │
│  特化抽取列（高频查询/校验，从 spec 提升为列）                           │
│  • data_type     (property 专用：int/float/double/bool/enum/...)     │
│  • access_mode   (property 专用：r / rw)                             │
│  • event_level   (event 专用：info / alert / error)                  │
│  • call_mode     (service 专用：async / sync)                        │
│  • relation_type (relation 专用：derivedFrom / partOf / feeds ...)   │
└──────────────────────────────────────────────────────────────────────┘
        │ spec.unit (property)                        │ spec.source/target (relation)
        ▼                                            ▼
┌──────────────────────┐               ┌────────────────────────────────┐
│ thingmodel_unit      │               │ thing_feature（自引用）         │
│ (单位管理，已落地)    │◄──────────────│ relation.source/target.id      │
│ reference_count 维护  │  property 引用 │ 可指向同表 feature 或外部实体   │
└──────────────────────┘               └────────────────────────────────┘
```

### 1.1 关键设计抉择

| 抉择点 | 方案 | 理由 |
|--------|------|------|
| 统一表 vs 分表 | **统一 `thing_feature`** | 公共字段占 70%+；用户要求"四类都是特征"；关系自引用需同表。详见 [08 章 §2](./08-特征架构决策.md) |
| 差异字段承载 | `spec` JSON | 四类差异是结构化复杂容器（参数列表/枚举项/嵌套 struct/关系四元组），抽列则稀疏；JSON 一次成型。详见 [08 章 §2.2.4](./08-特征架构决策.md) |
| 是否抽特化列 | **抽 5 个高频列**（data_type/access_mode/event_level/call_mode/relation_type） | 这 5 个是列表筛选、校验前置检查的高频维度；放 JSON 内则无法 DB 索引/无法 FieldMask 局部更新。其余差异全留 spec |
| spec 用 JSON 还是 protobuf 结构化 message | **proto message `FeatureSpec` + oneof**，序列化后存 JSON 列 | 既要 DB 灵活存储，又要强类型校验：proto 层定义结构（编译期检查），DB 层存其 JSON 序列化（查询灵活）。参照 [11 章](./11-特征API与Proto设计.md) |
| identifier 与 code 关系 | **两者并存**：code=清单编码（P-RUN-0001），identifier=程序标识符（powerSwitch） | 镜像清单两列；code 用于人读溯源，identifier 用于程序引用（设备协议映射）。code 租户内唯一，identifier 租户内唯一 |
| 关系 source/target 如何存 | spec 内 `{kind:"feature"/"external", id/code, type}` | 不建 DB 外键（避免循环依赖、支持外部实体）；校验在应用层（见 [12 章](./12-特征后端实现设计.md)） |
| 多租户 | 沿用 `tenant_id`（系统预置 `tenant_id=0`） | 与 unit/dict 全表一致 |

## 2. 表结构：`thing_features`（统一特征表）

> 对应《特征清单》每一行：P-RUN-0001 开关状态、E-GEN-0001 设备上线、S-GEN-0001 开机……

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | uint32 | PK, auto_increment | 来自 `mixin.AutoIncrementId` |
| `feature_type` | enum | NOT NULL | 特征类型：`PROPERTY` / `EVENT` / `SERVICE` / `RELATION` |
| `code` | string | NOT NULL, UNIQUE(租户内), immutable | 清单编码，如 `P-RUN-0001` / `E-GEN-0001` |
| `identifier` | string | NOT NULL, UNIQUE(租户内) | 程序标识符，如 `powerSwitch` / `turnOn`（设备协议映射用） |
| `name` | string | NOT NULL | 中文名，如"开关状态""设备上线" |
| `name_en` | string | optional | 英文名 |
| `description` | string | optional | 说明 |
| `applicable_scope` | string | optional | 适用设备范围，如"冷机/锅炉"或"通用"（对应清单"适用设备"列） |
| —— 特化抽取列（高频筛选）—— | | | |
| `data_type` | enum | optional | property 专用：int/float/double/bool/enum/text/date/struct/array（其余类型为 NULL） |
| `access_mode` | enum | optional | property 专用：`R`（只读）/ `RW`（读写） |
| `event_level` | enum | optional | event 专用：`INFO` / `ALERT` / `ERROR` |
| `call_mode` | enum | optional | service 专用：`ASYNC` / `SYNC` |
| `relation_type` | string | optional | relation 专用：关系类型，如 `derivedFrom` / `partOf` / `feeds` / `dependsOn` |
| —— 差异容器 —— | | | |
| `spec` | json | optional | 四类特征的结构化约束（见 §3），JSON 序列化的 `FeatureSpec` |
| —— 治理 —— | | | |
| `is_enabled` | bool | default true | 来自 `mixin.IsEnabled` |
| `sort_order` | uint32 | default 0 | 来自 `mixin.SortOrder` |
| `tenant_id` | uint32 | default 0 | 来自 `mixin.TenantID`，0=系统全局 |
| `created_by/updated_by/deleted_by` | uint32 | optional | 来自 `mixin.OperatorID` |
| `created_at/updated_at/deleted_at` | timestamp | optional | 来自 `mixin.TimeAt` |

**Ent Schema 骨架**（风格对齐 `unit.go`）：

```go
// thing_features 物模型特征表（统一承载属性/事件/服务/关系）
type Feature struct{ ent.Schema }

func (Feature) Annotations() []schema.Annotation {
    return []schema.Annotation{
        entsql.Annotation{Table: "thingmodel_features", Charset: "utf8mb4", Collation: "utf8mb4_bin"},
        entsql.WithComments(true),
        schema.Comment("物模型-特征表（属性/事件/服务/关系统一）/ Thing model feature"),
    }
}

func (Feature) Fields() []ent.Field {
    return []ent.Field{
        // ===== 公共字段 =====
        field.Enum("feature_type").
            Comment("特征类型：PROPERTY/EVENT/SERVICE/RELATION / Feature type").
            NamedValues(
                "Property", "PROPERTY",
                "Event", "EVENT",
                "Service", "SERVICE",
                "Relation", "RELATION",
            ).
            Optional().Nillable(),

        field.String("code").
            Comment("清单编码，如 P-RUN-0001（不可变）/ Code, immutable").
            NotEmpty().Immutable().Optional().Nillable(),
        field.String("identifier").
            Comment("程序标识符，如 powerSwitch / Program identifier").
            NotEmpty().Optional().Nillable(),
        field.String("name").Comment("中文名 / Name (zh)").NotEmpty().Optional().Nillable(),
        field.String("name_en").Comment("英文名 / Name (en)").Optional().Nillable(),
        field.String("description").Comment("说明 / Description").Optional().Nillable(),
        field.String("applicable_scope").
            Comment("适用设备范围，如 冷机/锅炉 / Applicable device scope").
            Optional().Nillable(),

        // ===== 特化抽取列（高频筛选，从 spec 提升）=====
        field.Enum("data_type").
            Comment("property 数据类型 / Property data type").
            NamedValues(
                "Int", "INT", "Float", "FLOAT", "Double", "DOUBLE",
                "Bool", "BOOL", "Enum", "ENUM", "Text", "TEXT",
                "Date", "DATE", "Struct", "STRUCT", "Array", "ARRAY",
            ).
            Optional().Nillable(),
        field.Enum("access_mode").
            Comment("property 访问模式 R/RW / Property access mode").
            NamedValues("R", "R", "RW", "RW").
            Optional().Nillable(),
        field.Enum("event_level").
            Comment("event 级别 INFO/ALERT/ERROR / Event level").
            NamedValues("Info", "INFO", "Alert", "ALERT", "Error", "ERROR").
            Optional().Nillable(),
        field.Enum("call_mode").
            Comment("service 调用模式 ASYNC/SYNC / Service call mode").
            NamedValues("Async", "ASYNC", "Sync", "SYNC").
            Optional().Nillable(),
        field.String("relation_type").
            Comment("relation 关系类型，如 derivedFrom/partOf / Relation type").
            Optional().Nillable(),

        // ===== 差异容器（四类 spec 的 JSON 序列化）=====
        // 用强类型 FeatureSpec（proto oneof）作为 JSON 目标，与项目 menu.meta 等字段实践一致
        field.JSON("spec", &thingmodelV1.FeatureSpec{}).
            Comment("特征结构化约束（按 feature_type 解读）/ Structured spec by feature_type").
            Optional().Nillable(),
    }
}

func (Feature) Mixin() []ent.Mixin {
    return []ent.Mixin{
        mixin.AutoIncrementId{}, mixin.TimeAt{}, mixin.OperatorID{},
        mixin.IsEnabled{}, mixin.SortOrder{}, mixin.TenantID[uint32]{},
    }
}

func (Feature) Edges() []ent.Edge {
    return []ent.Edge{
        // 多对一：property 引用单位（弱关联，spec.unit_id 指向 unit.id，不建强外键避免循环）
        // 关系：relation 自引用不建 edge（source/target 在 spec 内，应用层解析）
    }
}

func (Feature) Indexes() []ent.Index {
    return []ent.Index{
        // 租户级唯一：code 与 identifier 各自租户内唯一
        index.Fields("tenant_id", "code").Unique().StorageKey("uix_thingmodel_feature_tenant_code"),
        index.Fields("tenant_id", "identifier").Unique().StorageKey("uix_thingmodel_feature_tenant_identifier"),

        // 按类型查询（最高频：左侧树选类型 → 右侧列该类型特征）
        index.Fields("tenant_id", "feature_type").StorageKey("idx_thingmodel_feature_tenant_type"),
        index.Fields("feature_type").StorageKey("idx_thingmodel_feature_type"),

        // 特化列筛选（property 按 data_type、event 按 level 等）
        index.Fields("feature_type", "data_type").StorageKey("idx_thingmodel_feature_type_datatype"),
        index.Fields("feature_type", "event_level").StorageKey("idx_thingmodel_feature_type_level"),
        index.Fields("feature_type", "call_mode").StorageKey("idx_thingmodel_feature_type_callmode"),

        // 适用设备范围筛选
        index.Fields("applicable_scope").StorageKey("idx_thingmodel_feature_scope"),

        // 常用筛选
        index.Fields("is_enabled").StorageKey("idx_thingmodel_feature_is_enabled"),
        index.Fields("tenant_id").StorageKey("idx_thingmodel_feature_tenant_id"),
        index.Fields("sort_order").StorageKey("idx_thingmodel_feature_sort_order"),
    }
}
```

> **关于 `spec` 用强类型 `FeatureSpec`**：Ent 的 `field.JSON` 支持强类型目标（项目 `menu.meta` 用 `&permissionV1.MenuMeta{}`、`audit_log.device_info` 用 `&auditV1.DeviceInfo{}` 均此模式）。本设计用 `&thingmodelV1.FeatureSpec{}`（proto oneof）作为 JSON 目标，Ent 自动 marshal/unmarshal，DB 存其 JSON 序列化。`FeatureSpec` 是 oneof（[10 章 §3](./10-特征参数与spec设计.md)），按 `feature_type` 取 property/event/service/relation 分支，编译期类型安全。这与项目"结构化优于动态化"的整体风格一致。

## 3. 四类特征的 `spec` JSON Schema

`spec` 是差异收敛点。四类特征各有结构，由 proto `FeatureSpec` oneof 定义（强类型），序列化为 JSON 存 `spec` 列。以下给出每类的 JSON 结构与示例。

### 3.1 PROPERTY（属性）spec

```jsonc
{
  "dataType": "double",          // 与抽取列 data_type 一致（冗余，便于 spec 自洽）
  "accessMode": "rw",
  "category": "setting",         // 属性类别：runtime/measurement/setting/rated/statistic/environment
  "unit": {                       // 单位引用（打通单位管理）
    "unitId": 12,                 // → thingmodel_unit.id
    "unitCode": "celsius",        // 冗余 code，便于无 join 展示
    "unitSymbol": "℃"
  },
  "constraints": {                // 数据类型约束（按 dataType 解读）
    "min": -20,
    "max": 60,
    "step": 0.5,
    "defaultValue": 25
  },
  "enumItems": [                  // dataType=enum 时：枚举项列表（值→描述）
    { "value": 0, "label": "停机" },
    { "value": 1, "label": "手动" }
  ],
  "boolLabels": {                 // dataType=bool 时：0/1 描述
    "false": "关", "true": "开"
  },
  "textMaxLength": 256,           // dataType=text 时：最大长度
  "structFields": [               // dataType=struct 时：子字段（递归，每项是一个迷你 property spec）
    { "key": "inlet", "dataType": "double", "unit": {"unitCode":"celsius"}, "constraints": {"min":0,"max":40} },
    { "key": "outlet", "dataType": "double", "unit": {"unitCode":"celsius"}, "constraints": {"min":0,"max":40} }
  ],
  "arraySpec": {                  // dataType=array 时
    "size": 10,                   // 最大长度
    "element": { "dataType": "double" }   // 元素类型（可为 struct，递归）
  },
  "isRated": false                // 是否铭牌额定参数（对应清单第五章，运行时不变）
}
```

**对应清单示例**（`P-RUN-0001` 开关状态）：
```jsonc
{ "dataType":"bool", "accessMode":"rw", "category":"runtime",
  "boolLabels":{"false":"关","true":"开"}, "constraints":{"defaultValue":0} }
```

### 3.2 EVENT（事件）spec

```jsonc
{
  "level": "alert",               // 与抽取列 event_level 一致：info/alert/error
  "outputParams": [               // 输出参数列表（每个是一个迷你 property spec，无 accessMode）
    { "key":"parameterName", "dataType":"text" },
    { "key":"currentValue", "dataType":"double", "unit":{"unitCode":"celsius","unitSymbol":"℃"} },
    { "key":"limitValue", "dataType":"double", "unit":{"unitCode":"celsius","unitSymbol":"℃"} }
  ],
  "triggerCondition": "value > limit",  // 触发条件说明（仅展示）
  "severity": 2                   // 严重度 1~5（可选，用于告警分级）
}
```

**对应清单示例**（`E-GEN-0004` 参数越限告警）：
```jsonc
{ "level":"alert",
  "outputParams":[
    {"key":"parameterName","dataType":"text"},
    {"key":"currentValue","dataType":"double","unit":{"unitCode":"celsius","unitSymbol":"℃"}},
    {"key":"limitValue","dataType":"double","unit":{"unitCode":"celsius","unitSymbol":"℃"}},
    {"key":"unit","dataType":"text"}
  ]}
```

### 3.3 SERVICE（服务）spec

```jsonc
{
  "callMode": "async",            // 与抽取列 call_mode 一致：async/sync
  "inputParams": [                // 输入参数列表
    { "key":"temperature", "dataType":"double", "unit":{"unitCode":"celsius","unitSymbol":"℃"},
      "constraints":{"min":-20,"max":60}, "required":true }
  ],
  "outputParams": [               // 输出参数列表
    { "key":"success", "dataType":"bool" },
    { "key":"historyData", "dataType":"array", "arraySpec":{"size":1000,"element":{"dataType":"struct"}} }
  ],
  "timeout": 30,                  // 超时秒数（sync 服务用）
  "description": "设定目标温度"
}
```

**对应清单示例**（`S-GEN-0003` 重启）：
```jsonc
{ "callMode":"async",
  "inputParams":[
    {"key":"delay","dataType":"double","unit":{"unitCode":"second","unitSymbol":"s"},
     "constraints":{"min":0,"max":3600,"defaultValue":0}}
  ],
  "outputParams":[{"key":"success","dataType":"bool"}] }
```

### 3.4 RELATION（关系）spec —— 本体语义

```jsonc
{
  "relationType": "derivedFrom",  // 与抽取列 relation_type 一致
  "source": {                     // 关系主体（起点）
    "kind": "feature",            // feature=同表特征 / external=外部实体
    "id": 105,                    // kind=feature 时：thing_feature.id
    "code": "P-HVAC-0023",        // 冗余 code
    "identifier": "realtimeCOP"
  },
  "target": {                     // 关系客体（终点）
    "kind": "feature",
    "id": 88,
    "code": "P-HVAC-0017",
    "identifier": "compressorLoadRate"
  },
  "cardinality": "manyToOne",     // 基数：oneToOne/oneToMany/manyToOne/manyToMany
  "properties": {                 // 关系自身属性（可选，如"推导公式""权重"）
    "formula": "realtimeCoolingCapacity / currentPower"
  },
  "directional": true             // 是否有向（derivedFrom 有向，partOf 有向，relatedTo 无向）
}
```

**本体关系类型规划**（`relation_type` 取值，源自建筑设备 ontology 实践）：

| relation_type | 含义 | 方向性 | 示例 |
|---------------|------|--------|------|
| `partOf` | 组成关系（整体-部分） | 有向 | 压缩机 partOf 制冷机组 |
| `feeds` / `suppliedBy` | 供需关系（能量/物质流） | 有向 | 冷冻水泵 feeds 空调末端 |
| `controls` / `controlledBy` | 控制关系 | 有向 | DDC controls 冷机 |
| `derivedFrom` | 派生关系（属性推导） | 有向 | 实时COP derivedFrom 制冷量÷功率 |
| `dependsOn` | 依赖关系（服务前置） | 有向 | 启动 dependsOn 自检通过 |
| `locatedIn` | 空间包含 | 有向 | 设备 locatedIn 楼层（外部实体） |
| `relatedTo` | 泛化关联 | 无向 | 备用泵 relatedTo 主泵 |

> `source`/`target` 的 `kind` 支持外部实体（`external`），此时用 `type`+`code` 指向未来 product/device/space 表，本期不强校验存在性。

## 4. 约束（应用层 + DB 层）

| 编号 | 约束 | 实现层 | 说明 |
|------|------|--------|------|
| F1 | `code` 租户内唯一、不可变 | DB unique + ent `Immutable()` | 程序溯源标识 |
| F2 | `identifier` 租户内唯一 | DB unique | 设备协议映射标识 |
| F3 | `feature_type` 必填 | DB NOT NULL | 区分四类 |
| F4 | 特化列与 feature_type 一致性 | Service 校验器 | property→data_type/access_mode 必填；event→event_level 必填；service→call_mode 必填；relation→relation_type+source/target 必填。其余类型的特化列应为 NULL |
| F5 | spec 结构与 feature_type 一致 | Service 校验器（按类型分派） | property 的 spec 必须含 dataType/accessMode；event 必须含 level/outputParams；详见 [12 章](./12-特征后端实现设计.md) |
| F6 | property 引用的 unit 必须存在且启用 | Service（查 unit_repo） | 兑现单位管理挂钩点；同时维护 unit.reference_count |
| F7 | relation 的 source/target 若 kind=feature，目标须存在 | Service | 表内自引用一致性 |
| F8 | dataType=enum 时 enumItems 必填且非空 | Service 校验器 | 枚举无项无意义 |
| F9 | dataType=struct 时 structFields 必填 | Service 校验器 | |
| F10 | 删除 property 时，若引用了 unit，unit.reference_count -1 | Service 事务 | 维护单位引用计数（C6 约束的前提） |
| F11 | 删除被 relation 引用的 feature 时，提示而非级联 | Service | 关系完整性：被引用的特征删除前提示存在关系 |

> F6/F10 是打通单位管理预留挂钩点的关键：单位管理 [02 章 C6](./02-数据模型设计.md) 约束"reference_count>0 禁止删除"依赖此处的计数维护。

## 5. 数据示例（取自清单）

### 5.1 属性特征（取通用属性 + 暖通专属）

| feature_type | code | identifier | name | data_type | access_mode | spec 摘要 |
|--------------|------|-----------|------|-----------|-------------|-----------|
| PROPERTY | P-RUN-0001 | powerSwitch | 开关状态 | bool | RW | `{category:runtime, boolLabels:{false:关,true:开}}` |
