# 08 · 特征架构决策

## 1. 背景

《物模型特征清单.md》定义了建筑机电设备物模型的**三维结构**：

- **属性（Property）**：描述设备状态/参数的数据项，含数据类型、访问模式、类别、单位。
- **事件（Event）**：设备主动上报的通知，含级别（信息/告警/故障）+ 输出参数。
- **服务（Service）**：平台可下发的指令，含调用模式（异步/同步）+ 输入/输出参数。

清单还隐含了**第四维——本体关系（Relation）**：设备与设备、设备与空间、属性与属性之间的关系（如"冷冻水泵 supplyBy 制冷机组""供水温度 derivedFrom 回水温度"）。用户明确指出"**属性、事件、服务、关系等都是一种特征属性**"。

物模型的属性在定义时需绑定单位。当前系统已落地[单位管理](./01-架构决策.md)（`thingmodel_unit` 字典 + 换算引擎），其设计文档明确预留了挂钩点：

> `unit.reference_count` 字段预留，本期恒为 0，**由未来 `thing_property` 维护**。

本设计即落地那个"未来的 thing_property"——但不止于属性，而是将四类实体统一为**特征（Feature）**。

## 2. 核心决策：统一特征表，而非四张分表

### 2.1 判定结论

> **属性 / 事件 / 服务 / 关系四类实体，统一为一张 `thing_feature` 表，用 `feature_type` 字段区分。各自的差异化约束收敛到一个 `spec` JSON 字段。**

### 2.2 判定依据

#### 2.2.1 公共字段占比分析

通读《特征清单》四类实体的定义，提取公共字段：

| 字段 | property | event | service | relation | 公共? |
|------|:---:|:---:|:---:|:---:|:---:|
| 编码（code/标识符） | ✓ P-RUN-0001 | ✓ E-GEN-0001 | ✓ S-GEN-0001 | ✓ | ✓ |
| 名称/中文名 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 标识符（identifier，程序用） | ✓ powerSwitch | ✓ deviceOnline | ✓ turnOn | ✓ | ✓ |
| 说明 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 适用设备范围 | ✓ 冷机/锅炉 | ✓ 制冷机组 | ✓ 空调末端 | ✓ | ✓ |
| 排序 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 启用/停用 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 租户隔离 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 审计字段 | ✓ | ✓ | ✓ | ✓ | ✓ |
| **小计公共字段** | | | | | **9 项** |

差异化字段：

| 差异 | property | event | service | relation |
|------|----------|-------|---------|----------|
| 数据类型/约束 | data_type, min/max/step, unit, enum 项 | — | — | — |
| 访问模式 | access (r/rw) | — | — | — |
| 类别 | category (runtime/measurement/...) | — | — | — |
| 事件级别 | — | level (info/alert/error) | — | — |
| 输出参数 | — | output params | — | — |
| 调用模式 | — | — | call_mode (async/sync) | — |
| 输入参数 | — | — | input params | — |
| 关系语义 | — | — | — | source/target/type/cardinality |

**结论**：公共字段占 9 项，差异化字段每类仅 1~3 组。公共部分占比如此之高（>70%），分四张表意味着 9 个公共字段重复定义 4 次、4 套 CRUD、4 套 proto——**收益为零，成本翻四倍**。

#### 2.2.2 用户意图对齐

用户原话"**属性、事件、服务、关系等都是一种特征属性**"——这是对"统一抽象"的明确诉求。统一表正是这一抽象的直接落地。

#### 2.2.3 关系（Relation）的统一价值

关系作为 `feature_type=relation` 纳入同一表，意味着**关系的 source/target 可以指向同表内的任意特征**（一个属性 derivedFrom 另一个属性、一个服务 dependsOn 另一个服务）。若分表，跨表关系需多对多中间表，复杂度陡增。统一表让关系天然成为"表内自引用"。

#### 2.2.4 为什么不是"主表+扩展子表"混合方案

考虑过第三种方案：`thing_feature` 主表（公共字段）+ 四张 1:1 扩展表（`thing_property_ext`/`thing_event_ext`/...）。**否决理由**：
- 每次查询需 JOIN 扩展表，四类查询路径不同。
- 差异字段数量少（每类 1~3 组），且天然是"结构化但不需 DB 约束"的 JSON 形态（如 enum 项列表、struct 子字段）——抽成列反而稀疏。
- 单位管理的 `factor/offset` 是结构化数值才抽列；这里的差异多是"参数列表/枚举项/关系描述"，本质是 JSON。

> **准则**：结构化的标量（code/name/access/level/call_mode）抽为独立列建索引；结构化的复杂容器（参数列表、枚举项、嵌套 struct、关系四元组）用 JSON。这与单位管理"`factor+offset` 抽列、`formula_expr` 留文本"的取舍逻辑一致。

### 2.3 与物模型标准的关系

| 平台/标准 | 属性 | 事件 | 服务 | 关系 |
|-----------|------|------|------|------|
| 阿里云 TSL | ThingProperty | ThingEvent | ThingService | —（无） |
| 华为云 IoTDA | 属性 | 命令 | 命令 | —（无） |
| 腾讯云 IoT | property | event | action | —（无） |
| GB/T 35102 | 数据点 | 事件 | 服务 | —（无） |
| **本体（Ontology）** | — | — | — | 关系（三元组） |

主流物模型标准只定义三维（属性/事件/服务），**关系属于本体论（Ontology）范畴**。本设计将关系作为第四类特征统一进来，是一次**物模型 + 本体论的融合**——好处是复用同一套 CRUD/校验/前端框架，代价是关系语义需在 spec 内自洽（见 [10 章](./10-特征参数与spec设计.md) 关系语义模型）。

## 3. 总体架构

完全复用单位管理验证过的 Contract-First 流水线与三层架构，**零新依赖**：

## 4. 与单位管理的协作（打通预留挂钩点）

单位管理文档（[01 章 §4](./01-架构决策.md)）明确预留了两个挂钩点，本期兑现其一：

| 挂钩点 | 单位管理预留 | 本期兑现方式 |
|--------|-------------|-------------|
| **`unit.reference_count`** | 字段预留，恒 0 | 本模块 Create property（含 unit 引用）时 +1，Delete 时 -1；启用单位管理 C6 约束"被引用禁止删除" |
| 单位选择器 | `ListByCategory` API 已就绪 | property 表单选单位时调用 `useListUnitsByCategory`（单位管理已实现的 hook） |

> 第二个挂钩点（设备数据归一化运行时）仍属未来，本期不实现。

## 5. 设计原则

1. **镜像单位管理，不发明轮子**：表结构、proto 骨架、repo/service 范式、前端 Splitter 布局直接复用单位管理已验证的模式，降低评审与维护成本。
2. **公共抽列、差异入 JSON**：9 个公共字段抽为独立列（可索引、可 FieldMask 局部更新）；四类差异化约束收敛到 `spec` JSON，由应用层校验器保证正确性。
3. **spec 即契约**：`spec` 的 JSON 结构（schema）是四类特征的契约，前端表单按 schema 动态渲染，后端按 schema 校验。schema 定义见 [09 章](./09-特征数据模型设计.md)、[10 章](./10-特征参数与spec设计.md)。
4. **种子数据即出厂默认**：通用属性/事件/服务、各设备专属特征、铭牌额定特征，由种子脚本从《特征清单》提炼生成，版本化管理。
5. **关系自洽于表内**：relation 的 source/target 用 `{type:"feature", id}` 或 `{type:"external", kind, code}` 表达，可指向同表特征或外部实体（设备/空间），不在 DB 层建外键（避免循环依赖）。

## 6. 非目标（本期不做）

- **产品物模型模板**（`product`）：将特征组装成产品模板，设备实例继承——下一期。
- **设备实例**（`device`）：真实设备注册、三元组绑定——下一期。
- **实时数据上报与归一化运行时**：仅提供 spec 约束，不构建流处理。
- **关系图谱可视化**（知识图谱）：本期 relation 只做定义与存储，图谱渲染（neo4j/D3）属未来。
- **特征版本化**（多版本共存）：本期特征无版本概念，更新即覆盖（未来产品引用后需引入版本）。
